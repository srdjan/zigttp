The northstar: one declarative semantics, and a per-build certifying compiler

Where "a spec that reads like code and is provably tied to the bytecode" leads if we follow the state of the art all the way. Grounded in WebAssembly's SpecTec, LLVM's Alive2, CompCert, and proof-carrying code — with one structural advantage those projects don't have, and zigts does.

This is the destination, not the next commit. The companion note (zigts-formal-spec-design.html) describes the pragmatic L2 design we can build now; this document describes what that design is an on-ramp to, so every near-term decision points the right way. The state of the art is not a single technique — it is the composition of three lines of work that each matured in the last few years, plus a fourth that defines the ceiling.

The northstar. A single declarative semantics for zigts from which the readable spec, the executable reference interpreter, and the SMT encoding are all generated — paired with a compiler that emits, on every build, an SMT-checked, signed refinement certificate proving that this build's bytecode implements that semantics. Because zigts has removed the features that force approximation, the check is complete, not bounded — CompCert-grade per-build assurance at Alive2-grade automation, with no proof assistant in the loop.

Status · 2026-06-25: Phase 1 (the L2 slice) shipped

Every piece of the trajectory's first phase now exists in the tree, so the on-ramp is real, not planned:

  • One source. semantics.zig carries each node's denotation and lowering as data — 10 IR nodes and 7 opcodes specified — reduced to a stable semanticsHash.
  • Five checking mechanisms in zigts spec-check: the comptime drift gate, the stack-effect arity check, the symbolic lowering proof (12 value proofs: 5 binop + 2 unop instances + the literals/identifier/ternary/call core, plus 1 fused-opcode refinement), the differential corpus (10/10 cases) that runs the registry's denotations against the real compiler's output, and the SMT equivalence check over the slice value theory.
  • Readable view. zigts spec-render generates the TypeScript-shaped docs/spec/semantics.spec.ts from the same table, drift-gated in CI (spec-render --check).
  • Signed receipt. zigttp spec-check emits a persistent-key kind=semantics JWS, self-verified before it is written and third-party verifiable against the published attest JWK.

What stays honest: the drift gate is still a named-alphabet size pin (not per-member SMT coverage), the eq opcode has no differential case because the restricted language emits strict_eq for ===, and the third-party file verifier is a convenience still to add (the receipt is already verifiable with the existing well-known JWK). Phase 2 (the SMT refinement check) follows below.

Status · 2026-06-26: Phase 2 (the SMT refinement check) — first slice shipped

The symbolic executor is no longer the only judge of equivalence. A fifth mechanism in spec-check lifts the same registry data to a semantic check: it encodes each obligation over the slice's value model and asks Z3 whether the two sides agree on all inputs. This is the Alive2 move — and it admits exactly what structural equality (mechanism 3) cannot: structurally different but semantically equal lowerings.

  • Value model, encoded. A pure, wasm-safe encoder (semantics_smt.zig) turns an RPN denotation into SMT-LIB2 over native Int/Bool sorts with light type inference. (A tagged value datatype was tried first and rejected: it returns spurious sat on algebraic laws when a free leaf may be a bool, so getInt of it is unconstrained junk. Native sorts avoid that; the tagged/poison model is deferred to when the heap enters — see the caveat below.)
  • Algebraic laws as spec. The registry declares laws — add commutativity, neg and not involution — structurally different on each side, so mechanism 3 cannot see them. Z3 proves all three; they are hashed into semanticsHash and rendered into the spec.
  • The value model is an abstraction, and that boundary is enforced. Numbers are encoded as unbounded mathematical integers (ℤ), which is an abstraction of the engine's actual number type (i32 promoting to f64 on overflow). That is sound only for laws insensitive to float rounding, overflow, NaN, and signed zero. Associativity of + is the canonical law that holds over ℤ but fails on the engine (a child can be a large mul-overflow float, and f64 add is not associative past 2^53), so it is deliberately not asserted — a soundness review caught it after an earlier draft shipped it as "proven", which is exactly the false-attestation this mechanism must not produce. The boundary is now documented at the law table and the encoder.
  • Injected solver, honest outcomes. The driver takes the solver as a function pointer, so std.process.Child never enters the freestanding/wasm analyzer. Only a genuine counterexample is a failure: sat → ZTS755 (the law is false), an ill-typed obligation → ZTS756 (a spec-authoring bug), and a solver that cannot decide or run (absent, broken mid-run, undecidable, unwritable scratch dir) → unproven, reported but non-fatal — the same outcome as no z3, so an environmental failure never masquerades as a value-equivalence counterexample. SMT is also opt-out: ZIGTTP_Z3=off disables it even where z3 is installed.
  • In the receipt. The signed kind=semantics JWS carries the SMT availability and proved/total counts (payload v3); unproven obligations show up as proved < total.

What stays honest: this is the slice's value theory (ints, bools, seven operators) over an integer abstraction — not yet the engine's real f64 (or the heap). "Replace the symbolic executor" (the original Phase 2 wording) became "subsume it, with a structural fallback" so spec-check still works with no solver installed. The SMT check currently covers the refinements and the algebraic laws; a faithful f64/NaN encoding, extending across every per-node lowering, and then the value/heap model, is the rest of Phase 2.

01The pillars

1

One source generates every view

WebAssembly SpecTec · PLDI 2024 · adopted as official Wasm tooling, March 2025 12

Wasm's standard required every feature to appear in formal semantics, prose, and the reference interpreter — three hand-synced artifacts that drifted. SpecTec made them one: from a single declarative DSL it generates the typeset spec, the prose pseudocode, and a meta-level interpreter, automatically deriving the algorithmic (executable) semantics from the declarative (relational) one. The generated interpreter passes 100% of the official conformance suite. The same group did the JS version — JISET/ESMeta mechanically extract an executable interpreter from the ECMAScript standard's structured prose 8. This is the elevation of our "Zig table renders TypeScript" choice into "one source, many generated views, zero hand-sync."

2

Per-build translation validation via SMT

Alive2 · PLDI 2021 3

Alive2 takes LLVM IR before and after a transform, encodes both into SMT, and asks Z3 whether the output refines the input — same result, no new undefined behavior, for all inputs — fully automatic, no changes to LLVM. It found 47 real bugs in LLVM's own test suite and 21 more in memory optimizations. "Refinement" is exactly the relation behind get_loc_add ≡ get_loc; add. This now runs in spec-check for the slice's value theory (mechanism 5): the symbolic executor is subsumed by an SMT-backed equivalence check that encodes each obligation over the value model and lets Z3 discharge eval(node) ⊇ run(lower(node)) — and the registry's algebraic laws, which the syntactic check could not. The northstar extends that encoding from the slice's ints-and-bools to the full value/heap model.

3

The certificate the consumer checks without trusting the compiler

Certifying compilers · Necula & Lee, PLDI 1998 · Foundational PCC, Appel 1011

A certifying compiler emits both code and a certificate of a property the code satisfies; the consumer checks the certificate independently, with no need to trust the producer or verify the compiler itself. That is precisely what a kind=semantics receipt should be — and zigts already owns the Ed25519 sign/verify half. The northstar makes the receipt genuinely proof-carrying: it ships the discharged refinement obligations (or their hashes) so a third party can re-run the SMT checks.

4

The mechanized ceiling

CompCert · CakeML 912

Fully mechanized verified compilers — end-to-end Coq/HOL proofs. This is L3, the ceiling we keep a path to but do not require day one. The detail worth internalizing: even CompCert does not prove every pass once-and-for-all; several backend passes are done by translation validation, and seL4's compilation to machine code was validated, not proven. Mixing "proven-once" with "validated-per-build" is not a compromise — it is what the gold standard itself does.

·

The road not taken (and why)

K framework / Matching Logic · KJS · JSCert / JaVerT 567

K derives a parser, interpreter, symbolic model checker, deductive verifier, and equivalence checker from one semantics — the maximalist "all tools from the semantics" vision, proven on JS by KJS (passes all 2,782 ES5.1 core tests, symbolically executable). We do not adopt it wholesale for two reasons. First, zigts already has its analyzer family hand-written in Zig; the goal is to tie them to a spec, not regenerate them in a different framework. Second, KJS, JSCert, and JaVerT formalize full JavaScript — which is exactly what makes them enormous. zigts is not full JavaScript, by design, and that is the whole advantage.

02The angle that is uniquely zigts's

Restrictions make bounded validation complete

Alive2 is bounded because LLVM has unbounded loops — it unrolls to a fixed depth and accepts incompleteness as the price. The full-JS formalizations are enormous because JS is enormous. zigts has already thrown away exactly the features that force both compromises: no while, no unbounded back-edges, no try/catch, no eval, no dynamic this.

The repo's thesis — restrictions unlock proofs — applies one level down. The same cuts that make handler verification decidable make the bounded translation validation complete, not approximate, for the bulk of the language. Alive2's bound is a ceiling it cannot see past; for zigts's core the "bound" is just the structural depth of the IR tree, so the refinement check is exhaustive. zigts can target CompCert-grade per-build assurance at Alive2-grade automation, with no proof assistant in the loop. No one gets to make that claim about real JavaScript. zigts can — because it is not real JavaScript, by design.

The same restrictions-to-proofs argument the project already makes at the handler level, applied to the engine's own lowering.

04Trajectory

Four phases, each shippable and each strictly stronger than the last. Phase 1 is complete; Phase 2's first slice has landed; the rest of Phase 2 (full coverage, then the value/heap model) and Phase 3 follow.

Phase 1 · shipped
L2 slice
Zig table, hand-rolled symbolic checker, rendered spec.ts, differential corpus vs the real compiler, one signed receipt. All landed (2026-06-25).
Phase 2 · slice shipped
SMT validation
Z3 refinement check (Alive2-style) over the slice's value theory: a pure SMT-LIB2 encoder (integer abstraction), three SMT-certified algebraic laws, injected solver with structural fallback, counts in the receipt. Landed 2026-06-26. Remaining: faithful f64/NaN model, full per-node coverage, then the value/heap encoding.
Phase 3
DSL-generated spec
Promote the Zig table to a declarative DSL; generate spec.ts, the reference interpreter, and the SMT encoding from it (SpecTec-style).
Phase 4 · ceiling
Coq export (L3)
Export the DSL and per-node obligations to a proof assistant for mechanized end-to-end correctness. Optional, research-grade.

05Honest caveats

Three things are genuinely hard, and the northstar is more credible for naming them.

The value/heap SMT encoding is the real cost

Alive2's memory-model encoding was its own CAV 2021 paper 4. NaN-boxing, object identity, and inline caches all need careful SMT encoding — that is where the weeks go in Phase 2.

The FFI boundary stays an oracle

Native zigttp:* modules cannot be symbolically executed. They keep stated contracts, referenced from the module registry, outside the proof boundary — the same honest line the L2 design draws.

And building "the DSL generates everything" is a multi-month investment. SpecTec is a serious piece of engineering, not a weekend. Phase 3 is where the northstar earns its name, and it should be entered with eyes open.
