zigts · design note · rigor L2

A formal spec for zigts that reads like TypeScript and is provably tied to the bytecode

The meaning of every IR node and every bytecode opcode is written down once, as a declarative rule table, rendered as high-level TypeScript for people, and machine-checked against the real engine so the spec and the implementation cannot drift apart.

This note describes the design we converged on: substrate option (b) with a slice of (a) at L2 rigor. The authoritative semantics live as a Zig comptime table next to the enums they describe; TypeScript is a generated, readable projection of that table plus a small reference interpreter. "Provably tied" is realized by four concrete, layered mechanisms, not by prose.

01The gap this closes

zigts already treats correctness as hashed, signed, drift-proof artifacts. It has three declarative registries today, each the single source of truth for one facet of the system, each reducible to a stable hash. The proposal adds a fourth.

features
Surface grammar gate — which constructs are allowed or blocked, and the proof each cut unlocks.
json_diagnostics.zig
modules
The FFI surface — virtual-module exports, signatures, capabilities, laws.
builtin_modules.zig
rules
Diagnostics, reduced to policyHash() over all_rules.
rule_registry.zig
semantics NEW
What each NodeTag means and each Opcode computes, reduced to semanticsHash().
semantics.zig

The meaning of the language is the one facet with no declarative home. It exists only implicitly, scattered across two files: parser/codegen.zig (IR → bytecode lowering) and interpreter.zig (opcode execution). There is partial structure already — bytecode.zig defines OpcodeInfo {size, n_pop, n_push, name} and getOpcodeInfo(), a per-opcode table that sits right next to the Opcode enum — but it captures arity, not meaning. The semantics registry is, quite literally, "extend OpcodeInfo with what the opcode computes, add the NodeTag sibling, and hash the whole thing."

The result closes the loop end to end:

surface TS → IR · NodeTag → bytecode · Opcode → execution ⇒ signed kind=semantics receipt

02Where the semantics live, and why

The "reads like high-level TypeScript" requirement and the "provably tied" requirement pull in different directions, because at L2 there are three consumers of the spec and no single representation serves all three well.

1. The symbolic checker

A Zig program. To prove lower(node) == eval(node) for all inputs it needs the denotation stated relationally — reasoned over, not executed. It wants native Zig rules.

2. The drift gate

Trivial when the rule table sits beside the enum: reflection asserts the rule set equals the NodeTag/Opcode set; the build fails on any new opcode. A .ts source would need a generator and a reverse check.

3. People

Want it to read like TypeScript. That is a presentation requirement — met by rendering the table, not by authoring it in TS.

So the source of truth is Zig, and TypeScript is two derived views off it:

  • Source of truth: packages/zigts/src/semantics.zig — a comptime table binding each NodeTag/Opcode to its stack effect, denotation, and lowering template. Hashed into semanticsHash, signed.
  • Readable view: zigts spec renders the table to a literate spec.ts — the "reads like high-level TypeScript" artifact, generated so it can never drift from the source.
  • Executable companion: a thin spec/eval.ts in the zigts subset (dogfooded), arms 1:1 with the table, used for the differential pass.
Soundness bonus. Stating meaning independently in Zig gives the differential test a genuinely independent reference. Authoring the spec purely in the subset and running it on the engine would test the engine against itself — it would only catch divergence between the meta-circular and direct paths, never a shared mistake.

Honest trade-off: meaning is authored in Zig and read in TypeScript, not authored in TS. We judged the drift-proofing and symbolic-proof ergonomics worth that.

03The three layers

Each registry entry carries three facets, one per layer of the pipeline.

packages/zigts/src/semantics.zig — one entry, all three layers 
pub const semantics = [_]NodeRule{
    .{
        .tag = .binary_op,

        // Layer B - denotation: the meaning, as a relation over Values.
        // ⟦x⟧ is "the Value that subexpression x denotes".
        .denote = "applyBinop(op, ⟦left⟧, ⟦right⟧)",

        // Layer C - lowering template: this node -> this opcode sequence.
        // ⟨x⟩ is "the opcodes that evaluate x and leave its Value on the stack".
        .lower = "⟨left⟩ ⟨right⟩ <binop-opcode>",
    },
    // ... one entry per NodeTag. Layer A (surface -> IR) is the existing
    // `features` registry: every allowed construct maps to declared NodeTags.
};

Layer A

surface → IR. Every allowed feature lowers to declared NodeTags; coverage is total. Reuses the features registry.

Layer B

IR denotation. eval(node, env) -> Value, the reference big-step semantics. Value mirrors the NaN-boxed domain of value.zig.

Layer C

lowering + transition. Node → opcode sequence, plus a stack-machine rule per opcode. The L2 theorem lives here: eval(node) == run(lower(node)).

04The four binding mechanisms

"Provably tied" is the conjunction of four checks, ordered weakest-but-total to strongest-but-scoped.

Mechanism 1 — Table equality by construction

Reflection over the enum asserts the rule set is exactly the NodeTag and Opcode sets. Add an opcode without a rule and the build fails — the same comptime-validation trick that already guards the features array. This is what makes the spec impossible to leave behind. 

comptime {
    for (std.enums.values(Opcode)) |op| {
        if (!hasRule(op)) @compileError("opcode " ++ @tagName(op) ++ " has no semantics rule");
    }
}

Mechanism 2 — Stack-effect conformance by construction

Reuse OpcodeInfo.n_pop/n_push. The checker validates every lowering template is stack-balanced and that each transition rule honors the declared arity. A template that leaves the stack unbalanced is rejected before any semantic reasoning happens.

Mechanism 3 — Symbolic lowering correspondence L2 core

For each NodeTag, a symbolic executor proves that running lower(node) leaves the same denotation on the stack as eval(node). This is decidable because the IR is a tree and a single node's lowering has no back-edges — the same tractability the project already relies on ("no while means finite path enumeration"). The proof is structural induction; each per-node obligation is a single algebraic equality with the children's denotations held as symbols.

worked example: binary_op(add) 
// goal:  eval(add(L, R))  ==  run( lower(add(L, R)) )

// left side - Layer B denotation:
  eval(add(L,R)) = applyBinop(add, ⟦L⟧, ⟦R⟧) = numAdd(l, r)     // let l=⟦L⟧, r=⟦R⟧

// right side - Layer C, run the template on a symbolic stack:
  lower(add(L,R)) = ⟨L⟩ ⟨R⟩ add
    after ⟨L⟩  :  stack = [ l ]            // induction hypothesis on child L
    after ⟨R⟩  :  stack = [ l, r ]         // induction hypothesis on child R
    after add  :  stack = [ numAdd(l, r) ]  // opcode transition: pop b,a -> push numAdd(a,b)

// top of stack == eval(add(L,R)).  ∎  (modulo the IHs on L and R)

The optimized opcodes — inline caches (*_ic), superinstructions, fused arithmetic, type-specialized variants — are not proven from scratch. Each is proven as a refinement: its single-step transition equals the composition of the base opcodes it fuses. The proof then reduces to the already-proven core.

worked example: refinement get_loc_add ≡ get_loc; add 
// base transitions:
  get_loc i :  S          -> S ++ [ locals[i] ]
  add       :  S ++ [a,b] -> S ++ [ numAdd(a, b) ]

// composing them, starting from a stack that already holds one operand x:
  get_loc i ; add :  S ++ [x] -> S ++ [x, locals[i]] -> S ++ [ numAdd(x, locals[i]) ]

// superinstruction, single step:
  get_loc_add i   :  S ++ [x] -> S ++ [ numAdd(x, locals[i]) ]

// identical resulting stack, symbolically. The fused opcode is sound
// exactly when emitting get_loc; add would have been.  ∎

Mechanism 4 — Differential execution testing

A corpus — the examples/ handlers plus generated fixtures — runs through spec/eval.ts and through the real engine; observable outputs are asserted identical. This reuses trace.zig, --replay, and mock. It is testing, not proof: coverage-bounded, and labeled as such in the receipt.

05What is proven vs assumed

The whole point of L2 is to be precise about the strength of each claim. The receipt records exactly this ladder.

By construction
Table coverage (no drift) and stack balance. Enforced at compile time; cannot regress silently.
Symbolic (L2)
Lowering correspondence for the pure stack-deterministic core, plus refinement equivalences for every optimized opcode.
Differential
End-to-end agreement of reference vs engine over the corpus. Strong evidence, coverage-bounded — not a proof.
Assumed
Native module semantics (zigttp:*), GC/heap identity, and JIT tiers. Modeled as opaque oracles with stated contracts; referenced from existing harnesses, not re-proven here.

Non-goals: a fully mechanized end-to-end proof (that is L3 — export to an SMT solver / proof assistant); JIT correctness (rides the existing parity harness); the bodies of native modules (oracles).

06The artifact: a signed kind=semantics receipt

The spec becomes a first-class, hashable, signed, third-party-verifiable artifact — the same lifecycle as contracts and equivalence verdicts, reusing the Ed25519 attest keypair and appending a row to .zigttp/proofs.jsonl. 

{
  "kind": "semantics",
  "semanticsHash":   "9f3c8a…",   // SHA-256 of the canonicalized rule table (mirrors policyHash)
  "irTableHash":     "a1b2c3…",   // SHA-256 of NodeTag member names
  "opcodeTableHash": "c4d5e6…",   // SHA-256 of Opcode member names
  "corpusHash":      "77ef01…",   // differential corpus + results

  "opcodesProven":  96,

}

07CLI surface

Thin delegations into the same code path the other machine tools use — reachable as both zigts and zigttp.

zigts spec-render [--out p] [--check p]Render the literate spec.ts from the registry; --check is the CI drift gate against the committed docs/spec/semantics.spec.ts.
zigts spec-check [--json]Run all four mechanisms (drift gate, stack-effect, symbolic lowering, differential corpus). Exit 0 conform / 1 divergence (with a ZTS75x counterexample) / 2 error.
zigttp spec-checkSame checks, plus emits the persistent-key-signed kind=semantics receipt to .zigttp/semantics-receipt.jws (keyless zigts emits none).
zigts spec-hash [--json]Print semanticsHash for CI assertions (mirrors describe-rule --hash).

Diagnostic family ZTS75x: uncovered node/opcode, unbalanced lowering, lowering-divergence counterexample, refinement divergence. A third-party file verifier (zigttp verify <receipt>) is a still-to-add convenience; the receipt already verifies against the published attest JWK.

09Optional validating PoC slice

Per the house rule (validate at small scale before scaling up), a thin end-to-end slice proves the whole pipeline before the full tables are populated.

  • ~10 NodeTags: lit_int, lit_bool, identifier, binary_op, unary_op, ternary, if_stmt, return_stmt, block, call
  • ~15 Opcodes: push_i16/push_const/push_true/push_false, get_loc, add/sub/mul, lt/eq, if_false, goto, ret, call
  • The table-equality check runs against the full enums (fails on any uncovered member) — proving the drift gate works even though only the slice carries denotations.
  • Symbolic lowering proof for the slice + one signed kind=semantics receipt over the partial table.

Verification

  • zig build test for the two new modules.
  • zigts spec-check exits 0 on the slice; injecting a deliberately wrong lowering template yields a ZTS7xx counterexample (exit 1).
  • zigttp verify validates the emitted receipt.
  • Differential: run examples/ through both spec/eval.ts and the engine; outputs match.
