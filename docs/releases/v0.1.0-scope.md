# v0.1.0 Scope

What's in the first tag and what is deliberately out, so users evaluating the
runtime can size it against their needs without reading the source.

## In Scope

**HTTP server.** Threaded HTTP/1.1 with bounded URL, query, header, and body
limits; ETag-aware static files; canonical-path checks that defeat traversal
and symlink escapes; per-request runtime isolation via `HandlerPool`.

**JS engine.** Two-pass compiler (parser to IR to bytecode), pre-execution
bytecode verifier, NaN-boxed values, polymorphic inline caches with
megamorphic recovery, tiered JIT, generational GC, and request-scoped arenas.

**JS subset.** ES5 plus arrows, template literals, destructuring, spread,
`for...of`, optional chaining, nullish coalescing, `match`, `assert`, the pipe
operator, and typed arrays. Disallowed surface includes classes, async/await,
regex, `null`, `==`, `var`, `while`, `switch`, `this`, `new`, and `try/catch`.
See [Feature Detection](../feature-detection.md).

**Virtual modules.** 22 modules across data, HTTP, net, platform, security, and
workflow APIs: env, crypto, router, auth, validate, decode, cache, sql, service,
fetch, websocket, io, durable, compose, scope, url, id, http, log, text, time,
and ratelimit. Each declares required capabilities; sensitive SDK helpers and
runtime callbacks enforce those declarations at call time.

**Compile-time analyzers.** Contract extraction with handler properties, type
checker, flow checker with data labels, fault coverage, intent extraction,
system linking, strict ZigTS, and author-declared `Spec<...>` / `Effects<...>`
proof surfaces.

**Expert mode.** `zigttp expert` ships with compiler-in-the-loop edit veto,
Route Forge, Spec Forge, witness-aware repair planning, session persistence,
and proof-carrying patch payloads.

**Attestation.** Default-on Ed25519 JWS over `(bytecode_sha256,
contract_sha256, policy_sha256)`, embedded in self-extracting binaries, served
on `/.well-known/zigttp-attest`, and verifiable with `zigttp verify <url>`.

**Local deploy.** `zigttp deploy` emits a self-contained binary plus a JSONL
proof receipt. No network, Docker, or credentials required.

**Durable workflows.** `zigttp:durable` step recording, crash recovery,
sleep/sleepUntil/waitSignal primitives, and deterministic replay backed by an
on-disk oplog.

**Proof tooling.** Proof studio, terminal proof-card HUD, proof ledger, ratchet command
for spec monotonicity, contract diffing, behavior proof, system linking, proof
gate, witness corpus, and proof passport demo.

## Deferred

**Hosted cloud deploy.** `zigttp deploy --cloud`, account login/logout, grants,
and remote review flows are outside the beta. Core deploy is local-only.

**Evented I/O backend.** Zig 0.16's evented networking surface is not mature
enough for the release target. v0.1.0 ships the threaded HTTP server.

**Multipart parser.** Handlers receive raw bodies. `multipart/form-data`
parsing is deferred to a future module.

**JIT code-cache eviction.** Native code is freed on handler swap. A bounded
LRU for long-running servers is deferred.

**Tier-2 JIT stability.** Optimized loop-specialized code paths exist and pass
tests, but are not the stability promise for v0.1.0.

**WebSocket peer close-code validation.** RFC 6455 frame parsing is present;
explicit validation of peer-sent close codes is deferred. See the
[WebSocket module reference](../virtual-modules/net/websocket.md).

## Supported Platforms

- macOS on Apple Silicon and Intel.
- Linux on x86_64 and arm64.
- Windows is not supported in v0.1.0.
- Zig toolchain: 0.16.0 stable.

## After the beta: follow-up fixes

### sql-schema in the expert apply path

The interactive `zigttp expert` agent cannot land a `zigttp:sql`-backed handler.
Its edit-apply path (`pi_apply_feature_plan` -> `edit_simulate` -> `precompile.runCheckOnly`)
never threads a `sql_schema_path`, so `validateSqlContractNative` (packages/tools/src/precompile.zig:2768)
rejects any handler with SQL queries: `error.MissingSqlSchema` ("zigttp:sql queries
require --sql-schema"). The compiler veto is therefore unreachable for SQL handlers
authored through the agent, regardless of code quality.

Fix outline:
- Thread a project-level `sqlSchema` (already parsed into `runCheckOnly` opts as
  `sql_schema_path`, precompile.zig:485) through the expert tools:
  `zigts_expert_edit_simulate`, `pi_apply_feature_plan`, `pi_apply_repair_plan`.
  Source it from `zigttp.json` (project_config) so the agent inherits it like `dev`/`serve` do.
- Have `zigttp init` scaffold a starter `schema.sql` and reference it in `zigttp.json`,
  so a fresh project's SQL handlers verify out of the box.
- Until then, agent-authored persistence demos should use `zigttp:cache`
  (cacheGet/cacheSet, no schema) and guard `cacheGet` for `undefined`.

Discovered 2026-05-31 while recording the todolist demo. Separate from the six
expert-loop fixes that landed the same day (signature discoverability, UTF-8-safe
request serialization, gzip decompression, error surfacing).
