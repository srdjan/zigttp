# zigttp v0.1.0 Beta - Benchmark Results

Raw results and reproduction methodology for the v0.1.0-beta performance pass.
This is the evidence behind the Performance section of
[`v0.1.0-beta-checklist.md`](v0.1.0-beta-checklist.md). Numbers here are
measured, not estimated.

## Toolchain note - `zig build bench` (resolved)

During this pass `zig build bench` failed on Zig 0.16.0 stable with
`embedded_handler_stub.zig: error: file exists in modules 'zigts' and 'zigts0'`.
The cause: the `bench` exe's `embedded_handler` import wired the Debug `zigts`
module while its root module was the ReleaseFast `runtime_bench_dep`, so the
graph carried two `zigts` instances. `build.zig` now wires the bench exe's
`embedded_handler` import to a matching ReleaseFast `zigts` module, and
`zig build bench` works with no `-Doptimize` flag. The numbers in section 1
below were captured before the fix with the `-Doptimize=ReleaseFast` workaround;
the result is identical either way (the bench exe is ReleaseFast regardless).

`zig build bench -- --json`. ReleaseFast. Single run.
Total wall time across all 13 benchmarks: **52.7 ms**.

| Benchmark | Iterations | Time (ms) | Ops/sec |
|-----------|-----------:|----------:|--------:|
| intArithmetic | 50000 | 4.317 | 11,582,117 |
| stringConcat | 50000 | 2.572 | 19,440,124 |
| stringOps | 50000 | 4.890 | 10,224,948 |
| objectCreate | 50000 | 6.114 | 8,177,952 |
| propertyAccess | 50000 | 4.735 | 10,559,662 |
| arrayOps | 50000 | 4.695 | 10,649,627 |
| functionCalls | 50000 | 4.321 | 11,571,395 |
| recursion | 25 | 8.961 | 2,789 |
| jsonOps | 5000 | 1.534 | 3,259,452 |
| gcPressure | 50000 | 6.886 | 7,261,109 |
| httpHandler | 5000 | 0.854 | 5,854,800 |
| httpHandlerHeavy | 2000 | 1.873 | 1,067,805 |
| forOfLoop | 50000 | 0.974 | 51,334,702 |

Observations from the per-benchmark `perf` counters:

- **JIT tiering fires.** `functionCalls` promoted 3 functions (2 to baseline, 1
  to optimized-candidate); `recursion` promoted 2. No deopts, no deopt storms.
- **Inline caches work.** `propertyAccess` is near-monomorphic: 249,995 PIC
  hits to 5 misses.
- **One hidden-class hot spot.** `objectCreate` shows 100,001 PIC misses -
  hidden-class churn on fresh object creation, consistent with the open
  "Request object hidden-class churn" item in `TODO.md`.
- `gcPressure` ran clean (no deopts, no recoveries).

## 3. Cold start

Wall time from process launch (`Popen`) to the first complete HTTP response.
Fresh `zigttp serve -e <handler> -q --no-env-check` process per run, one
trivial JSON handler, tight TCP connect-poll loop (sub-millisecond
granularity). 20 measured runs after 3 warmups (binary warm in the page
cache). The measurement harness is a ~70-line Python script; per
CONTRIBUTING.md it belongs in the `zigttp-bench` sibling repo, not here.

| Pool size | min | p50 | p95 | max |
|-----------|----:|----:|----:|----:|
| 1 | 3.21 ms | 7.90 ms | 36.15 ms | 47.33 ms |
| 4 | 3.46 ms | 7.23 ms | 10.03 ms | 11.18 ms |
| 28 (default) | 3.97 ms | 7.34 ms | 8.10 ms | 8.99 ms |

The default pool gives the tightest tail. Pool size 1 has two clear outliers
(36 ms, 47 ms) - scheduling jitter or lazy first-slot allocation under no
prewarm.

**Beta gate (p50 and p95 under 5 ms): not met.** p50 ~7.3 ms and p95 ~8.1 ms
sit ~2-3 ms above the gate. This is a narrow miss, not a structural one.

**Reconciled with `docs/performance.md` (2026-05-22).** That doc previously
stated ~71-103 ms cold start with a "dyld 80-90 ms" breakdown - roughly 10x
these measurements, and self-contradictory (it also listed dyld at 15-20 ms).
A ~3.5 ms floor disproves an 80-90 ms dyld cost on current hardware. Follow-up
runs on a busier host showed cold start is variance-dominated (p50 ~7-15 ms,
tail to ~60 ms under scheduling contention), which is the likely origin of the
old ~71 ms figure. `docs/performance.md` now states the measured distribution
and the stale breakdown was removed.

## 4. Baseline memory (RSS)

Resident set size sampled immediately after the first response, before any
sustained load.

| Pool size | RSS |
|-----------|----:|
| 1 | ~13.38 MB |
| 4 | ~13.76 MB |
| 28 (default) | ~13.78 MB |

RSS is ~13.4-13.9 MB and **essentially independent of pool size** - the
baseline is loaded code plus base structures, not pool slots.

**Beta gate (under 5 MB RSS): not met.** Measured ~13.4 MB minimum. The
README's old ~4 MB claim is not reproduced by this method; the README and
`docs/performance.md` now state the measured ~13 MB.

## 5. HTTP throughput - zigttp vs Node.js vs Deno

Trivial JSON handler (`{"ok":true}`) on each runtime, identical load:
`hey -n 300000 -c 50` after a 50,000-request warmup, loopback, same machine,
each runtime in its default configuration. Every run was verified: all 300,000
responses HTTP 200, zero transport errors.

| Runtime | Version | Req/sec | p50 latency | p99 latency |
|---------|---------|--------:|------------:|------------:|
| zigttp | 0.1.0-beta | 112,393 | 0.20 ms | 2.00 ms |
| Deno | 2.7.14 | 113,058 | 0.40 ms | 0.90 ms |
| Node.js | 26.0.0 | 86,927 | 0.60 ms | 1.20 ms |

zigttp and Deno are level on throughput (~112-113k req/s, within run-to-run
noise); both are ~30% ahead of Node.js on this workload. zigttp has the lowest
median latency (0.20 ms) but the widest tail (p99 2.00 ms vs Deno's 0.90 ms) -
the same scheduling tail seen in the cold-start measurement. Threading models
differ and are left at defaults: Node runs a single event loop, `Deno.serve`
is multi-threaded, zigttp uses a 28-runtime worker pool. Pool size barely moved
zigttp throughput (28 vs 64 slots: 110.7k vs 112.0k req/s).

**Methodology note.** `hey`'s duration mode (`-z`) produced an unstable run
against zigttp - a `-z 10s` test ran ~30 s wall and reported a deflated ~38k
req/s - while behaving normally against Node and Deno. Fixed-count mode (`-n`)
was stable for all three and is used above. The `-z` interaction is worth a
look as a possible keep-alive quirk; it does not affect the fixed-count
numbers.

**Reconciled with `docs/performance.md`.** That doc previously had zigttp at
79,743 RPS, 0.76x Deno. This pass measures zigttp at parity with Deno
(~0.99x); `docs/performance.md` now carries the table above.

## Reproduction

```bash
# 1. Engine microbenchmarks (JSON to stdout)
zig build bench -- --json

# 2. Regression check vs benchmarks/perf-baseline.json
zig build bench-check

# 3. Release binary for cold start / RSS
zig build -Doptimize=ReleaseFast

# 4. Cold start + RSS: launch `zigttp serve -e <handler> -p <port> -q
#    --no-env-check` per run, time Popen -> first HTTP 200, sample RSS via
#    `ps -o rss=`. Harness belongs in the zigttp-bench repo.

# 5. HTTP throughput: trivial JSON handler per runtime; `hey -n 300000 -c 50`
#    after a 50k warmup. Use an absolute path to the Node binary - an nvm
#    lazy-load shell stub shadows `node` in non-interactive shells. Node and
#    Deno handler equivalents and the runner belong in the zigttp-bench repo.
```
