# zigttp v0.1.0 Beta - Pre-Release Checklist

Launch-readiness gate for the zigttp public beta. zigttp is a Zig-based
serverless JavaScript runtime for FaaS, powered by the `zigts` engine.

This document tracks whether the beta is ready to ship. It is a readiness gate,
not a release-mechanics runbook. The mechanical "cut the release" steps (test
gates, cross-compile verification, tagging, GitHub Release) live in
[`RELEASE_CHECKLIST.md`](../../RELEASE_CHECKLIST.md) at the repo root and are
not duplicated here.

## How to use this checklist

- Every item is a checkbox. Each carries a `Status:` note from a codebase
  survey (2026-05-22) describing what already exists, so a checked box means
  the work is genuinely done, not assumed.
- Status terms:
  - `EXISTS` - already implemented; this item is verify-only.
  - `PARTIAL` - exists but incomplete; finish it.
  - `GAP` - does not exist; create it.
  - `MEASURE` - a measurement or audit task with no fixed artifact.
- Any unchecked launch blocker must have a named owner and a disposition
  (ship / fix / defer) recorded before v0.1.0 ships. Use the Disposition
  column in the Launch blockers table at the bottom.

## 1. Performance & Benchmarking

- [ ] Measure cold starts from process launch to first handler response; beta gate: p50 and p95 under 5ms on target hardware.
  - Status: MEASURED 2026-05-22 (Apple M4 Pro, macOS 26, ReleaseFast, `zigttp serve -e`). Cold start is variance-dominated: ~3.5ms floor, p50 ~7-15ms depending on host load, tail to ~60ms under scheduling contention - see [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md) and the reconciled [`performance.md`](../performance.md). The 5ms p50/p95 gate is not met. Box stays unchecked until the gate is met or a documented exception is recorded.
- [ ] Measure baseline memory before the first request; beta gate: under 5MB RSS, or a clearly documented exception.
  - Status: MEASURED 2026-05-22. RSS ~13.4MB right after the first response, essentially independent of pool size - see [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md). The 5MB gate is missed; the README and `performance.md` now state the measured ~13MB (the old ~4MB claim is gone). Box stays unchecked pending a documented exception or a footprint reduction.
- [ ] Add repeatable benchmark scripts for the `/health`, `/echo`, JSON-response, and error-response paths.
  - Status: PARTIAL. `zig build bench` (`packages/runtime/src/benchmark.zig`) covers 12 microbenchmarks including an HTTP handler; per-path scripts to be confirmed or added in the `zigttp-bench` sibling repo.
- [x] Compare equivalent handlers against Node.js and Deno with identical payloads, concurrency, and hardware.
  - Status: DONE 2026-05-22. `hey -n 300000 -c 50`, a trivial JSON handler on each runtime: zigttp 112,393 req/s, Deno 113,058, Node.js 86,927 - all 200s, zero errors. zigttp is at parity with Deno and ~30% ahead of Node. See [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md).
- [ ] Stress test the `zigts` JIT under FaaS-style burst concurrency.
  - Status: PARTIAL. The JIT is a real multi-tier compiler (baseline + optimized, x86-64 and ARM64) and `benchmark.zig` already emits tier-promotion and deopt counters; a burst-concurrency harness is still needed.
- [ ] Stress test the generational GC with short-lived request objects and mixed payload sizes.
  - Status: GAP. The GC is generational (bump-allocated nursery + tenured mark-sweep with SIMD sweeping); a dedicated GC stress test does not exist.
- [ ] Record throughput, latency, cold start, warm start, RSS, allocator failures, and GC pause metrics.
  - Status: PARTIAL. Cold start, RSS, engine-microbenchmark throughput, and HTTP throughput/latency (req/s, p50, p99) are recorded in [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md). Still missing: warm-start, allocator-failure, and GC-pause metrics.

  - Status: DONE. [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md) records the environment, exact commands, the microbenchmark table, the 0.00%-regression verdict, cold-start and RSS results, and reproduction steps. ## 3. Security & Reliability

- [x] Audit request isolation across pooled handler executions.
  - Status: DONE. Audit completed. Each request checks out an isolated runtime; on release the value stack, call depth, exception state, catch depth, and the request arena are reset, and `consumed_body_objects` / `last_request_body_len` are cleared. The only state shared across requests is immutable handler bytecode and module state a virtual module deliberately persists (for example `zigttp:cache`). No request-scoped data (headers, body, query, locals) is shared. Arena-escape audit counters (`arena_audit_failures`, `persistent_escape_failures`) fire under `std.debug.runtime_safety`.
- [x] Add regression tests proving no headers, body data, locals, or module state leak across requests unless explicitly shared.
  - Status: DONE. Added `test "HandlerPool does not leak request data across pooled requests"` in `packages/runtime/src/zruntime.zig`. It runs alternating tainted (body + `X-Secret` header + marked url) and clean (no body, no header) requests through a size-1 pool - forcing reuse of one recycled runtime - and asserts the clean response carries none of the prior request's body, header, or url across four cycles. Passes under `zig build test-zruntime` (210/210).
- [x] Verify Zig VM memory safety around bytecode execution, stack bounds, allocator lifetimes, and error paths.
  - Status: DONE. Verified the bounds checks in `packages/zigts/src/context.zig`: `push` returns `StackOverflow` when the 1 MB value stack is full, `pushFrame` returns `CallStackOverflow` at the 1,024-frame call cap, `pushCatch` returns `CallStackOverflow` at the catch-handler cap; the interpreter's saved-state stack caps at 1,024. All raise typed errors rather than corrupting memory.
- [x] Run sanitizer or equivalent debug builds where available.
  - Status: DONE. The Zig Debug build is the sanitizer-equivalent here - it carries bounds, overflow, and illegal-behavior safety checks. The full suite runs green under it: `test-zigts` 1090, `test-zruntime` 210, `test-cli` 407. Added `test "Context stack overflow surfaces typed errors"` in `packages/zigts/src/context.zig` exercising all three engine stack caps (value, call, catch).
- [x] Enforce server payload size limits with documented defaults.
  - Status: DONE. The 1MB `max_body_size` default is documented in [`reliability.md`](../reliability.md), and an over-limit `Content-Length` now returns `413 Payload Too Large` on both server backends, enforced from the header value before the body is buffered - covered by `test "threaded handleSingleRequestSync returns 413 for oversized body"` in `packages/runtime/src/server.zig`. The limit is fixed in the binary (no CLI flag).
- [x] Add basic rate limiting, or explicitly document why it is deferred from v0.1.0.
  - Status: DONE (documented deferral). A "Rate limiting" section in [`reliability.md`](../reliability.md) records that v0.1.0-beta has no server-level rate limiting and why: in FaaS/edge targets the platform owns network-layer limits; a standalone `zigttp serve` should sit behind a proxy; application-level limits belong in the handler via `zigttp:ratelimit`. The known consequence (pool saturation for an unproxied standalone server) is stated.
- [x] Redact secrets and request bodies from default logs.
  - Status: DONE. Audited default request-path logging across `server.zig`, `zruntime.zig`, `live_reload.zig`, and `contract_runtime.zig`. The default logs never print request bodies, header values, query strings, or env var values - only HTTP method, path, and status, plus error type names. Required env vars are logged by name, never value. Secure by default; no redaction layer needed.
- [x] Define crash behavior, panic handling, and non-zero exit expectations.
  - Status: DONE. [`reliability.md`](../reliability.md) specifies it - handler and engine faults fold into `HandlerError` and return HTTP 500 with the server staying up; there is no custom panic handler (Zig default, abort on a bug); CLI exit codes are 0 success / 1 error / 2 bad-input, listed per command.

## 4. Developer Experience & CLI

- [x] Ensure JS/TS syntax errors include file, line, column, snippet, and an actionable message.
  - Status: DONE.   - Status: DONE. - [x] Confirm the CLI exits with stable non-zero codes for check, build, and test failures.
  - Status: DONE. Verified against the code: `check` exits `1` on analyzer errors and `0` clean; `build` exits `1` on any failure and `0` on success; `test` exits `1` on test failures, `2` on an invalid fixture, `0` when all pass. Matches the exit-code table in [`reliability.md`](../reliability.md).
