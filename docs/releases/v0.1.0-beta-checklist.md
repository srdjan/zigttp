# zigttp v0.1.0 Beta - Pre-Release Checklist

Launch-readiness gate for the zigttp public beta. zigttp is a Zig-based
serverless JavaScript runtime for FaaS, powered by the `zigts` engine.

This document tracks whether the beta is ready to ship. It is a readiness gate,
not a release-mechanics runbook. The mechanical "cut the release" steps (test
gates, cross-compile verification, tagging, GitHub Release) live in
[`RELEASE_CHECKLIST.md`](../../RELEASE_CHECKLIST.md) at the repo root and are
not duplicated here.

## How to use this checklist

- Every item is a checkbox. Each carries a `Status:` note from a codebase
  survey (2026-05-22) describing what already exists, so a checked box means
  the work is genuinely done, not assumed.
- Status terms:
  - `EXISTS` - already implemented; this item is verify-only.
  - `PARTIAL` - exists but incomplete; finish it.
  - `GAP` - does not exist; create it.
  - `MEASURE` - a measurement or audit task with no fixed artifact.
- Any unchecked launch blocker must have a named owner and a disposition
  (ship / fix / defer) recorded before v0.1.0 ships. Use the Disposition
  column in the Launch blockers table at the bottom.

## 1. Performance & Benchmarking

- [ ] Measure cold starts from process launch to first handler response; beta gate: p50 and p95 under 5ms on target hardware.
  - Status: MEASURED 2026-05-22 (Apple M4 Pro, macOS 26.4.1, ReleaseFast, `zigttp serve -e`, 20 runs). p50 7.3ms, p95 8.1ms - see [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md). The 5ms gate is narrowly missed (~2-3ms over), nowhere near the ~71ms `docs/performance.md` states; that ~10x discrepancy is unresolved. Box stays unchecked until the gate is met or a documented exception is recorded.
- [ ] Measure baseline memory before the first request; beta gate: under 5MB RSS, or a clearly documented exception.
  - Status: MEASURED 2026-05-22. RSS ~13.4MB right after the first response, essentially independent of pool size - see [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md). The 5MB gate is missed; the README's ~4MB claim is not reproduced by this method. Box stays unchecked pending a documented exception or a reconciled measurement.
- [ ] Add repeatable benchmark scripts for the `/health`, `/echo`, JSON-response, and error-response paths.
  - Status: PARTIAL. `zig build bench` (`packages/runtime/src/benchmark.zig`) covers 12 microbenchmarks including an HTTP handler; per-path scripts to be confirmed or added in the `zigttp-bench` sibling repo.
- [x] Compare equivalent handlers against Node.js and Deno with identical payloads, concurrency, and hardware.
  - Status: DONE 2026-05-22. `hey -n 300000 -c 50`, a trivial JSON handler on each runtime: zigttp 112,393 req/s, Deno 113,058, Node.js 86,927 - all 200s, zero errors. zigttp is at parity with Deno and ~30% ahead of Node. See [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md).
- [ ] Stress test the `zigts` JIT under FaaS-style burst concurrency.
  - Status: PARTIAL. The JIT is a real multi-tier compiler (baseline + optimized, x86-64 and ARM64) and `benchmark.zig` already emits tier-promotion and deopt counters; a burst-concurrency harness is still needed.
- [ ] Stress test the generational GC with short-lived request objects and mixed payload sizes.
  - Status: GAP. The GC is generational (bump-allocated nursery + tenured mark-sweep with SIMD sweeping); a dedicated GC stress test does not exist.
- [ ] Record throughput, latency, cold start, warm start, RSS, allocator failures, and GC pause metrics.
  - Status: PARTIAL. Cold start, RSS, engine-microbenchmark throughput, and HTTP throughput/latency (req/s, p50, p99) are recorded in [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md). Still missing: warm-start, allocator-failure, and GC-pause metrics.
- [x] Publish benchmark methodology and raw results so landing-page claims are reproducible.
  - Status: DONE. [`v0.1.0-beta-benchmarks.md`](v0.1.0-beta-benchmarks.md) records the environment, exact commands, the microbenchmark table, the 0.00%-regression verdict, cold-start and RSS results, and reproduction steps. It also documents that `zig build bench` fails on Zig 0.16.0 stable and needs the `-Doptimize=ReleaseFast` workaround.

## 3. Security & Reliability

- [ ] Audit request isolation across pooled handler executions.
  - Status: EXISTS. A HandlerPool with arena-escape audit counters exists (`packages/runtime/src/runtime_pool.zig`); this item is the audit itself.
- [ ] Add regression tests proving no headers, body data, locals, or module state leak across requests unless explicitly shared.
  - Status: GAP. No explicit data-leakage test exists.
- [ ] Verify Zig VM memory safety around bytecode execution, stack bounds, allocator lifetimes, and error paths.
  - Status: EXISTS. Bounds checks are in place - the checked value-stack push returns `StackOverflow`, with 1024-deep call and state-stack caps (`packages/zigts/src/context.zig`).
- [ ] Run sanitizer or equivalent debug builds where available.
  - Status: PARTIAL. The Zig Debug build carries safety checks; run an explicit pass and add a stack-overflow test (listed under "Missing Tests" in `TODO.md`).
- [x] Enforce server payload size limits with documented defaults.
  - Status: DONE (documented), with a finding. The 1MB `max_body_size` default is now documented in [`reliability.md`](../reliability.md). Findings: the limit is fixed in the binary (no CLI flag), and an over-limit body makes the server close the connection with no response - it does not return `413 Payload Too Large`. The missing `413` is in Launch blockers.
- [ ] Add basic rate limiting, or explicitly document why it is deferred from v0.1.0.
  - Status: PARTIAL. The `zigttp:ratelimit` virtual module exists (handler-level); a server-level per-IP or global gate is a gap.
- [ ] Redact secrets and request bodies from default logs.
  - Status: MEASURE. Needs an audit of default logging.
- [x] Define crash behavior, panic handling, and non-zero exit expectations.
  - Status: DONE. [`reliability.md`](../reliability.md) specifies it - handler and engine faults fold into `HandlerError` and return HTTP 500 with the server staying up; there is no custom panic handler (Zig default, abort on a bug); CLI exit codes are 0 success / 1 error / 2 bad-input, listed per command.
