What it is

A JavaScript runtime where the compiler proves your code before it ships

zigttp is a pure-Zig JavaScript engine paired with a serverless HTTP runtime for FaaS workloads. It targets AWS Lambda, Azure Functions, Cloudflare Workers, and the edge. The design goals are instant cold starts, a small binary, request isolation, and zero dependencies. You write a single handler function; the compiler extracts a contract from it, derives a least-privilege sandbox, and verifies a set of properties at build time - so a deploy is something you can hand someone a receipt for.

One binary

The runtime, the engine, and your handler compile into a single self-extracting executable. No Docker image, no node_modules, no runtime to install on the host.

Per-request isolation

Each request gets an isolated runtime from a pre-warmed pool. State cannot leak across requests, and the compiler proves it rather than asking you to trust it.

The compiler is the product

The interesting work happens at build time: contract extraction, type and flow checking, fault coverage, and signed proof receipts - not just bundling.

The core idea

Restrictions are not limitations - they are proofs

zigttp runs an opinionated subset of TypeScript: the footguns are removed. Each cut is deliberate, and each one buys a specific soundness guarantee the analyzer could not otherwise make. Take away unbounded loops and you can enumerate every path; take away exceptions and every return is visible to the type system. The proof card in zigttp dev renders this table against your actual handler, so you see exactly which restriction earned each green chip.

See it fail in real time. Drop a Date.now() into a handler that declares deterministic and the chip lights up red with a Why: row pointing at the source line. Wrap it in step("ts", () => Date.now()) from zigttp:durable and the chip flips back green. The full cut-to-proof map is available at the terminal with zigttp restrictions.

Get started

From zero to a proven, running handler

Install the binary, scaffold a project, and you are editing a handler with live proofs in under a minute. Every command after install auto-detects the project from zigttp.json, so the steps take no arguments.

  1. Install zigttp

    Pre-built binaries for macOS and Linux (x86_64 and aarch64). One line:

    $ curl -fsSL https://raw.githubusercontent.com/srdjan/zigttp/main/install.sh | sh

    Prefer a tarball? Grab one from GitHub Releases. Or build from source with Zig 0.16.0: 

    $ git clone https://github.com/srdjan/zigttp.git && cd zigttp
$ zig build -Doptimize=ReleaseFast

  2. Scaffold a project

    Create a project and step into it. A new project is a handler, a manifest, a starter test, and a static dir:

    $ zigttp init my-app && cd my-app

my-app/
  zigttp.json            # entry + port
  src/handler.ts         # your handler fn
  tests/handler.test.jsonl
  public/                # static assets
  README.md

    Templates: --template basic (default), api, or htmx (JSX, src/handler.tsx).

  3. Build the handler with the expert agent

    zigttp expert is a compiler-in-the-loop coding agent - the way to write a new handler or reshape the template one. Describe the goal and the proof obligations the result must meet; it proposes edits and the compiler re-proves after each until they hold. Point it at a provider key first, then describe what you want: 

    $ zigttp auth claude     # store an Anthropic key (or export ANTHROPIC_API_KEY)
$ zigttp expert
  goal: add GET /api/users/:id, proven no_secret_leakage
  re-proving after each edit ... 4/4 discharged - apply? [y/N]

    See the full loop in The expert agent below.

  4. Run with live proofs

    Start proof-aware live reload on http://localhost:3000. Edit src/handler.ts and the proof card rebuilds on every save - see Writing a handler and the proof model. 

    $ zigttp dev
  listening on http://localhost:3000
  watching src/handler.ts - proving on save

  5. Run the tests

    The starter ships a declarative fixture - request in, expected response out, no test framework to wire up.

    $ zigttp test
  tests/handler.test.jsonl - 1 passed

  6. Deploy a signed binary

    Build, prove, and emit a self-contained signed binary under .zigttp/deploy/. No cloud, Docker, or network - see Deploy + verify. 

    $ zigttp deploy
  wrote .zigttp/deploy/my-app  (signed)

Writing a handler

A handler is one function: Request in, Response out

No framework, no router class, no lifecycle hooks. You export a handler function. Start minimal, then reach for virtual modules and an author-declared Spec<...> as the handler grows.

The JavaScript / TypeScript subset

TS and TSX files run directly through a native type stripper. The unsupported features below are detected at parse time with a suggested replacement - they are the cuts that earn the proofs above.

● Supported

  • arrow functions
  • function declarations
  • for...of (arrays)
  • if / else
  • match expression
  • destructuring
  • spread
  • template literals
  • optional chaining
  • nullish coalescing
  • typed arrays
  • array HOFs
  • Object.keys/values/entries
  • range()
  • TS / TSX / JSX
  • ES6 modules

● Not supported (with suggestions)

  • class
  • async / await
  • Promise
  • this / new
  • try / catch
  • throw
  • var
  • while
  • switch
  • for...in
  • null
  • == / !=
  • ++ / --
  • regex
  • delete

Use undefined as the sole absent-value sentinel. Response helpers: Response.json(), .text(), .html(), .redirect().

The strict profile narrows further to a Canonical Normal Form: one spelling per operation - an if or match instead of a ternary, x = x + 1 instead of +=. zigttp normalize rewrites a handler into that form automatically, and a handler with zero such diagnostics carries the canonical proof chip.

Virtual modules

Batteries, implemented in Zig, governed by capabilities

Import platform functionality from zigttp:* modules. They are implemented natively in Zig - not JS shims - and each declares the capabilities it needs (clock, crypto, random, network egress). Those declarations are enforced at call time and feed the handler's least-privilege sandbox, so importing zigttp:sql is what grants database authority, and nothing else gets it. 

import { jwtVerify, parseBearer } from "zigttp:auth";
import { sql, sqlOne } from "zigttp:sql";
import { parallel, race } from "zigttp:io";

Author-declared proofs

Declare what must hold, in the return type

Intersect your handler's Response return type with Spec<...> to name the properties it must satisfy. The compiler either discharges every obligation or fails the build - a declared property that no longer holds is a compile error, not a runtime surprise. Spec is a type-only import, so it strips away cleanly and costs nothing at runtime. 

import type { Spec } from "zigttp:types";

function handler(req: Request): Response & Spec<"idempotent" | "deterministic"> {
  // the compiler now proves both properties on every path
  return Response.json({ ok: true });
}

deterministic

Same input, same output. No ambient clock, randomness, or unrecorded I/O on any path.

no_secret_leakage

Values labelled secret or credential never reach the response body.

injection_safe

User input cannot flow unescaped into a sink that would interpret it as code or query.

idempotent

Re-running the handler produces no additional observable state change.

state_isolated

Nothing written during one request is visible to another. The pool guarantees it; the compiler checks it.

canonical

The handler is in Canonical Normal Form: one canonical spelling per operation. Run zigttp normalize to rewrite a handler into it.

Compiler-in-the-loop

The expert agent: the model proposes, the compiler decides

zigttp expert is a coding agent built around the analyzer, and the primary way to develop a handler in zigttp - whether you are writing one from scratch or evolving the template that init scaffolded. You give it a goal and the properties the result must satisfy; it proposes an edit, and the compiler re-proves the handler after every change. The loop runs until each obligation is discharged or the agent reports it cannot get there. What lands is something the compiler has already proven - not just something that looked plausible.

The compiler drives convergence

A property-goal autoloop: the model proposes an edit, the analyzer re-proves it, and the loop repeats. Progress is measured by the proof result, not the model's confidence.

You stay in control

Review and approve or reject each edit interactively. Pass --yes to auto-apply only the edits that pass verification.

Every edit is provable

Each applied edit is checked for behavioral equivalence and logged as a signed kind=equivalence receipt in .zigttp/proofs.jsonl.

Fits your workflow

Interactive by default; --print for one-shot and --mode rpc for CI. Sessions persist - pick up with --resume. Backed by Claude or OpenAI.

Idiomatic by construction

A draft that slips into a non-canonical form - a ternary, a += - is auto-normalized to its canonical spelling and applied in one shot, instead of bouncing back for a retry.

Deploy + verify

A deploy you can hand someone a receipt for

zigttp deploy auto-detects the handler and project name, verifies the handler, and emits a self-contained binary at .zigttp/deploy/<project>. No credentials, Docker, or network. Proof receipts ship default-on and travel inside the binary.

Verifiable from any machine

Every fresh build signs its receipt and embeds it. Deploy, then verify the running service from anywhere - no shared secret, just the embedded public key: 

$ zigttp deploy
  wrote .zigttp/deploy/my-app  (signed)

$ zigttp verify https://my-app.example.com
  signature OK - contract + bytecode + policy

What every receipt signs

A JWS over three hashes, signed with a persistent Ed25519 identity generated on first use:

  • the extracted contract
  • the compiled bytecode
  • the derived sandbox policy

The running server returns Zigttp-Proofs and Zigttp-Attest headers on every request, and serves the full envelope plus the public JWK at /.well-known/zigttp-attest.

Build from source. Validated on Zig 0.16.0 stable. zig build produces the three binaries (debug); zig build -Doptimize=ReleaseFast for release; zig build run -- examples/handler/handler.ts -p 3000 to run a handler directly. Hosted cloud deploy is deferred from this beta - the self-contained binary is the supported path.
