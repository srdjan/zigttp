HATEOAS + HAL workflows

The server doesn't just return data. It returns the data plus the set of actions you're allowed to take next. The client follows those, instead of hardcoding URLs. That's the whole idea, and it makes a workflow drive itself.

1. HATEOAS: the state machine lives on the server

HATEOAS stands for Hypermedia As The Engine Of Application State. Strip the jargon and it's one rule:

Every response carries links to the next valid actions. The client starts at one URL and discovers everything else by following links it was handed, not by knowing them in advance.

Think of a restaurant menu. You don't memorize the kitchen's internal order numbers; you read what's offered right now and point at it. A breakfast menu and a dinner menu are the same restaurant in different states, offering different choices. A HATEOAS API works the same way: the available actions are a function of server state, and the server hands them to you on every response.

The payoff:

  • The client can't reach an illegal state. If "pay" isn't offered, the UI literally has no button for it. No client-side rule duplicating server logic.
  • URLs stay the server's business. Routes can move; clients follow relations (pay, cancel), not paths.
  • Workflows become discoverable. An automated client can walk the graph by following relations it understands.

2. HAL: a tiny JSON convention for the links

HATEOAS is the idea; HAL (Hypertext Application Language, media type application/hal+json) is one concrete way to write it down. A HAL document is just your normal JSON with up to three reserved keys:

_links Navigation. A map keyed by relation (rel). Every resource has a self link; others point to related resources you can GET.
_embedded Nested resources shipped inline to save round-trips (e.g. an order's line items), each itself a HAL document.
_templates Write actions, from the HAL-FORMS extension: method, title, and input properties. This is what zigttp emits for non-GET affordances.
// A HAL representation of order #42 in the "pending" state
{
  "id": 42,
  "total": 1999,
  "status": "pending",
  "_links": {                              // where can I GO?
    "self":     { "href": "/orders/42" },
    "customer": { "href": "/customers/7" }
  },
  "_templates": {                          // what can I DO?
    "pay": {
      "method": "POST", "target": "/orders/42/pay",
      "title":  "Pay now",
      "properties": [{ "name": "card" }, { "name": "cvc" }]
    },
    "cancel": { "method": "DELETE", "target": "/orders/42", "title": "Cancel" }
  }
}

Same idea for a human: render those _links as <a> tags and the _templates as <form>/buttons, and a browser walks the exact same state machine a service does.

4. Following links is the "engine"

      

        An automated workflow does exactly what you just did by hand: fetch a
        resource, find an affordance by its relation, invoke it, repeat.
        It never builds a URL:
      

      
// generic HATEOAS client loop — no URL is ever hardcoded
let order = await get("/orders/42");             // entry point

while (order._templates?.pay) {              // is "pay" offered?
  const t = order._templates.pay;
  order = await invoke(t.method, t.target, {  // follow it
    card: "4242...", cvc: "123"
  });
}
// now order.status === "paid", and pay is gone from _templates


      
In zigttp this is a first-class primitive

      

        zigttp's follow(resource, rel, init?) from
        zigttp:workflow resolves an affordance by its
        relation and dispatches to the proven route, so an orchestrator advances
        the workflow by naming relations, never paths:
      

      
import { follow } from "zigttp:workflow";

function handler(req) {
  const order = getOrder(42);          // { status: "pending", ... }
  const paid = follow(order, "pay", {     // follow the "pay" affordance
    body: JSON.stringify({ card: "4242..." })
  });
  return follow(paid, "ship");          // then the "ship" affordance it now exposes
}


      
5. zigttp's twist: one declaration, two renderings, proven

      

        The example above is the manual JSON. In zigttp you don't hand-write
        _links/_templates.
        You declare the affordance set once with the
        resource() global, and the runtime renders it
        for whoever asked:
      

      
function handler(req: Request): Response {
  const order = { id: 42, total: 1999, status: "pending" };

  return resource(order, {
    self:   { href: "/orders/42" },
    pay:    { href: "/orders/42/pay", method: "POST", title: "Pay now",
              target: "#order", swap: "outerHTML" },
    cancel: { href: "/orders/42", method: "DELETE", title: "Cancel" },
  });
}

      

        

          Accept: application/hal+json
          

            A service gets the HAL document: data, plus
            _links for the GET/no-method affordances
            and _templates for the write ones
            (fields become
            properties).
          

        

        

          Accept: text/html
          

            A browser gets an HTMX fragment from the same affordances:
            <a> for navigation,
            <button hx-post>/hx-delete
            carrying hx-target/hx-swap.
          

        

      

      

        Two more things zigttp adds on top of plain HAL:
      

      
    
        
  • One source of truth. The same affordance set drives HAL for machines and HTMX controls for humans, so the two can never drift apart.
    • 
        
  • Affordances are proven at compile time. The contract extractor records every emitted affordance, and the system linker checks each one resolves to a real route. A follow(order, "ship") that points at a relation no handler serves is a build failure, not a 404 in production.
    • 
      


      

        The shape to remember: data + the next legal moves, every
        time. HAL is how you write the moves down; HATEOAS is letting
        them drive the workflow; zigttp's resource()
        lets you declare them once and have the compiler prove they connect.
