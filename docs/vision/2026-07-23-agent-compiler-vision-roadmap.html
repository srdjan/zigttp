1. The category thesis

An agent-compiler is a single system in which three artifacts are engineered against each other: a restricted language whose safety and correctness properties are decidable; a proof system that is total over that language; and a code-generating agent whose entire authoring loop — goal, feedback, repair, and reward — is expressed in the proof system's vocabulary rather than in natural language. The defining property: the set of programs the agent can emit converges on the set of programs the compiler can prove. The agent does not write code and then check it. It authors inside a fence, and the fence is load-bearing for everything else.

This is categorically different from the two things it will be mistaken for.

Not "an LLM plus a linter"

A linter is heuristic, advisory, incomplete, and post-hoc — suggestions over an unbounded language that a model routes around, and "no findings" means nothing. The zttp verifier is the opposite: the subset bans the constructs that make analysis undecidable (while, try/catch, async, and the class/this/new family), so the analyzer's structural walk over the IR terminates and covers every path exhaustively rather than sampling (docs/verification.md:18-27) — that is what makes "no findings" a theorem, signed with Ed25519. Precise scope: this is a property of the analysis, not a claim that compiled execution is acyclic — codegen still emits loop back-edges for for-of, and recursion is legal (bounded at runtime by a stack cap, which is not a termination proof; recursive capsules are explicitly treated as unproven). The exhaustiveness that matters for the reward is the analyzer's, and it holds.

Not "a compiler with an LLM bolted on"

In that architecture the compiler is a passive oracle: the agent generates into an unbounded space, gets error strings, and searches blindly. Here the compiler actively structures the search. It supplies the goal as typed data (typed holes: expected type, in-scope bindings, capability budget, undischarged Spec obligations). It supplies feedback as typed data (the rule registry maps diagnostics to a typed RepairIntent — ~43 of the registry's ~67 rules carry one). It executes a growing fraction of repairs with no model call at all — though "has a typed intent" and "applies deterministically" are different metrics: of the 18 RepairIntent variants the shared executor (repair_apply.zig) currently lowers ~7 to an actual source edit (Phase B model-free apply). And it grades the outcome with a total, deterministic verdict that doubles as a training reward.

Why zttp is uniquely positioned

Four properties that took years of deliberate restriction to earn and that a general-purpose stack cannot retrofit:

  1. Decidability by construction. The subset was restricted on purpose so correctness is decidable; every restriction maps to a proof it unlocks (zttp restrictions, docs/restrictions-to-proofs.md). A V8-based competitor would have to un-ship JavaScript to get here.
  2. A cheap, deterministic, non-flaky reward channel. The verifier is total over the subset, deterministic, runs in milliseconds, and compiles to a hermetic wasm64 module (zig build wasm, analyzer_only; wasm64 because NaN-boxing needs 64-bit pointers, so the host needs memory64) — a low-variance reward oracle. Scope it honestly: the exported wasm path grades a single handler with null SQL-schema and system-contract paths, so SQL, linked systems, and standalone-app tasks need an extended or native reward path, and the throughput ("millions per hour") is a target to benchmark, not a measured number. General RLHF reward models are learned approximations that get hacked; this one is a proof with a versioned soundness history. Two honest limits keep this from being "perfect": the verifier has carried soundness bugs (a taint fail-open lived 14 review passes before its 2026-07-10 fix; JIT-promoted faults are a documented open item), so the training pipeline must version the verifier and relabel on every soundness fix; and even a sound verifier rewards vacuous satisfaction — a handler that returns 501 everywhere is fully green — so the reward must always pair the proof with functional intent (see §2b, §6).
  3. A partly comptime-anchored spec. The rule registry is comptime-assembled and the semantics registry is gated five ways, including z3 SMT equivalence (mechanism 5) and an exclusion audit that machine-refutes over-claims — so a defined slice of the reward's meaning is proven. Be precise about the boundary: the six rule-metadata tables are hand-authored (only the flow table has an enum-exhaustiveness gate today), SMT covers scoped value-level obligations with unknown non-fatal, and the formal spec explicitly excludes native-module, GC, and JIT correctness. Proven obligations and tested assumptions are different tiers; the doc should never conflate them.
  4. Low output entropy per idiom. normalize plus the canonical-form property (ZTS612-620) collapses a chunk of the output distribution — nine specific patterns (ternaries, compound assignment, spreads, templates, params, destructuring, boolean comparison). Identifier choice, decomposition, and API selection stay free, so this is measured entropy reduction, not one surface form per idiom — but it is still what makes a small model plausible and canonical examples reusable. (Quantify its effect on token entropy and first-draft rate before leaning on it for model-size selection.)
The moat The moat is not the agent, the model, or the proof system individually — each can be imitated. It is the closed data loop none of them can exist without the other two: every expert session emits verifier-labeled, signed, replayable training data for a language whose corpus exists nowhere else. A competitor needs the language to get the verifier, the verifier to label the data, and the data to train the model. Replicating any one piece buys nothing.

2. The three pillars — a triangle, not a list

2a. The skilled agent — from prompted model to compiler-native loop

Today the agent is a well-prompted remote model (expert_persona.zig ~128 KiB system prompt; expert_workflow.zig task classification; 50+ tools; module signatures fetched live) wrapped in a deterministic veto/repair loop (veto.zig edit-simulate dry-runs; repair_plan.zig typed plans; auto_repair.zig retry blocks). That already moved first-draft-correct from 3/6 to 7/7. The next stage inverts the relationship: the compiler stops being scaffolding around generation and becomes the medium of generation.

  • Typed holes become the native unit of work (ideation survivor 1). hole()/todo() plus check --holes --json turns "write me a handler" into a sequence of machine-readable completion contracts. The loop (loop.zig) stops asking for whole files and starts asking for hole fills — a radically smaller generation problem with a locally checkable answer.
  • Certified forms shrink authoring to parameterization (survivor 6). resource()/webhook()/proxy() comptime generators mean the common 80% of handlers is "choose a form, fill its holes," with output re-proven by the full pipeline — no trusted-template loophole. With the combinator algebra (survivor 3), most sessions never require open-ended synthesis.
  • The veto/repair loop goes mostly deterministic. RepairIntent coverage grows toward totality; every diagnostic with a unique canonical fix is applied with no model call. The model is invoked only where genuine choice exists — which also defines the residual task the local model must learn.

2b. The local zts model

Why local. The veto loop is millisecond-scale; a remote round-trip is seconds and dollars. The solo-developer persona works offline, on a laptop, on private code whose contracts include its secrets topology (flow labels, env vars, egress hosts) — exactly the material you least want streamed to a third-party API. The tight hole-fill → veto → repair loop is only viable when inference is local. Remote frontier models stay in the system as teacher and fallback, routed by task class via the existing classify() machinery.

The data flywheel: a self-labeling corpus — with an eligibility gate. The infrastructure already emits the raw material, but the raw stream is not training-clean and must pass a filter before it is a correctness label. Two facts to respect: veto approval means only zero new violations (EditOutcome.ok = new_count == 0) — pre-existing violations can remain, and a verified_patch record is written with post_apply_ok as a field, not a gate; and the provenance chain is a plain SHA-256 fingerprint, not a signature (equivalence receipts are best-effort and absent for new files). So the pipeline needs a separately signed training-eligibility envelope — zero total violations, post_apply_ok, required functional tests, a verified proof receipt, and the reward-environment manifest (§2c) — before a record counts as a positive label; weaker records become delta or negative examples. With that filter, the sources are:

The label is only as good as the eligibility gate and the verifier version that produced it — which is exactly why §2c makes reward-environment identity load-bearing.

Training, staged. (1) SFT / distillation of an open 3-8B code model on a fill-in-the-middle objective (aligned with holes). The synthetic channel needs care: naive "enumerate canonical forms, mutate, verify" produces (program, green) pairs that teach valid syntax, not task intent — the trivially-correct noise to avoid. The two channels that carry real signal are reverse-diagnostics (invert a RepairIntent on a green handler to synthesize a red one, then train on red + diagnostic → verified green edit; the label is the verifier's own diagnostic — dense and correct) and the expert session ledgers (persisted verified_patch events, which carry intent but whose volume is the cold-start dependency). (2) Preference tuning on veto-rejected/repaired pairs, balanced per rule family. (3) RLVR with the proof as reward: rollout = fill a hole or draft against a spec; reward = green proof (hard gate) + provenPathRatio partial credit + per-property discharge shaping + functional-test pass fraction − veto penalties − a length penalty. Critical: the property floor is fixed from the task spec, not the generated code — otherwise a policy inflates provenPathRatio by emitting handlers that track fewer properties. The hermetic wasm64 analyzer is the strategic asset: a ms-latency, parallel reward oracle.

Honest caveat — the small-model bet is three bets, not one "Narrow grammar ⇒ a 3-8B model wins" is a hypothesis, and it splits by task class with very different confidence. Repair (44 rules carry a typed RepairIntent; Phase B model-free apply already ships with no model) — high confidence, arguably needs no model at all. Typed hole-fill — medium confidence, directly testable, because the context is constrained decoding. Whole-handler-from-intent — low confidence: the bottleneck is deciding what the handler should do, and grammar narrowing does not help reasoning; keep the frontier model as proposer with verifier reranking here. classify() routes the reasoning-heavy turns to the frontier teacher; the deterministic scaffolding shrinks how often they occur. The honest failure mode is the middle band — too novel for the small model, too frequent to remote-route economically. State the bet as falsifiable and let the eval (fixed suite, small vs frontier, cost per verified handler; the pi codegen eval baseline moved 3/6 → 7/7) decide it before committing to RLVR.

2c. The proof system — correctness spine and reward signal

The proof system carries two loads: the user-facing guarantee, and the label generator for pillar (b). Both demand the same thing first — soundness of the reward. A concrete, instructive example: a taint-laundering fail-open lived through 14 review passes before it was closed on main (4e46e143 for member/optional/arrow-call laundering, 00af8391 for the JSX channel, 2026-07-10). While open, that fail-open would have poisoned training — a verifier that labels a leaky program green certifies the bug class into both SFT labels and RLVR reward. The durable lesson is not "fix that one bug" (done) but a process obligation: the training pipeline versions the verifier and relabels banked data on every soundness fix, so a corpus is never trusted across a soundness boundary. Critically, policyHash alone cannot gate this — it hashes rule metadata, not checker implementations, and it was recorded unchanged across the very soundness fix above (00af8391). The pipeline needs a reward-environment manifest — analyzer artifact SHA, source commit, compiler/semantics/policy/module hashes, dataset schema, and an explicit soundness epoch — and must relabel against the artifact or epoch, never the policy hash. Survivor 2 (unforgeable Validated<T,"schema"> brands, minted only by validateJson, demanded by sinks) is now defense-in-depth for the same class: it converts the global taint dataflow question into a local type obligation a second, independent layer enforces.

Beyond that, the proof system deepens along three axes: coverage (route-table determinism, survivor 4 — whose "prove a property over finite declarative data" pattern is the exact prototype for the app transition tables in §3; plus the deferred SMT type-precondition layer); provenance (survivor 7 — receipts record the DAG of function-level capsules that discharged each claim, which is simultaneously the user audit story and the training data-lineage story); and totality of repair (every registry rule either has a deterministic RepairIntent or is explicitly classified as requiring judgment — the compiler formally delimiting the model's job).

3. The standalone-applications expansion

The hardest and most valuable move: growing zts from function handler(req): Response to standalone applications without surrendering the acyclic-CFG decidability the entire thesis rests on.

3.1 The principle: the loop lives in the runtime; the logic stays a tree

A standalone application differs from a handler in exactly one semantic respect: unbounded execution — it loops, waits, and reacts indefinitely. The naive move (admit while, admit events-as-callbacks) reintroduces arbitrary back-edges and destroys the tree-walk proof. The correct move is prefigured in the codebase: zttp:durable is a working precedent — not a general existence proof — that unbounded, resumable execution can be decomposed into individually analyzable steps. run()/step()/sleep()/waitSignal() drive a workflow through position-sensitive replay; the driver that re-enters the function is trusted Zig (durable_recovery.zig). Read the precedent honestly: today it is request-scoped, run() must return a Response, several exports (run/sleep/waitSignal/signal) are effectful outside step(), and replay switches to live execution on any event/call mismatch. So the standalone design borrows the shape (effects through named, journaled steps; the back-edge in the runtime, not the user's code) but still owes explicit semantics for event delivery, checkpoint atomicity, code/state versioning, idempotency, backpressure, cancellation, live (non-replayable) sockets, and replay divergence. That list is the real engineering cost, not a corollary.

Generalized: user code is a finite set of step functions f_i : (State_i, Event) -> (State_j, Effects), each an ordinary zts function that receives every existing proof (response totality becomes transition totality; plus fault coverage, flow labels, capability enforcement, cost envelopes). Note the precise claim: a step is input-bounded with a proven cost envelope, not literally acyclic — recursion is legal at runtime (guarded by stack-overflow limits in interpreter.zig) and array HOFs iterate runtime-sized arrays, so "the logic stays a tree" is shorthand for "the step terminates and its cost is a signed, input-relative property" (the in-flight Cost Contracts work is the honest form of this). Control structure is data — a finite transition table in the app contract, encoded per the house rule as a type capsule (App<{states,events,transitions}>), never a new keyword; states and events are enums, and a guard is a pure predicate over State discharged per-transition by the existing prover (guards are where verification effort will concentrate). The event loop is a trusted Zig supervisor: pump event, look up transition, acquire runtime, invoke step, persist state, release — the HandlerPool isolation and arena-per-request model transfer intact as arena-per-event.

Where the bridge relocates the hard part rather than removing it Three limits to state plainly, or the decidability claim is dishonest. Infinite state domain. The transition table is finite, but the reachable state space is not — a step that appends to a list makes it unbounded, and properties over infinite-state systems are not finite-graph model-checking. Decidability is claimed only for exact syntactic table properties (reachability over enum nodes, transition coverage, dead-state absence) plus the per-step properties the prover already discharges. Two corrections: a guard that is a predicate over unbounded State turns "can this edge fire" into an infinite-state satisfiability question, not graph traversal — so guarded reachability needs inductive state invariants or schema-bounded State (via schemaCompile), and bounded model-checking gives bounded evidence recorded in the receipt, not the theorem. And "supervisor termination" is the wrong liveness goal for a loop meant to run forever — replace it with per-event progress, bounded per-event handling, and deadlock/fairness obligations, some runtime-monitored rather than statically proven. Finite ≠ tractable. States × events × guards can still explode, so the design caps declared cardinality and falls back to bounded model-checking with an explicit coverage report rather than silently timing out. Effects, not just control. "Restart = replay" is inherited from durable only for journaled effects; durable replay is position-sensitive and works precisely because every effect is forced through a named step(). A live socket read, wall clock, or push is not replayable — so the supervisor must impose the same all-effects-through-journaled-handlers discipline, and that discipline is the real engineering cost of the standalone story, not a free corollary of the durable analogy.

3.2 Entrypoints and packaging — by ascending runtime risk

  1. cli() — one-shot: argv/stdin in, structured output out. No event loop; a handler with a different transport. Ships first because it exercises multi-entrypoint contracts and packaging with zero new runtime machinery.
  2. worker() — a queue consumer: the existing zttp:queue receive/ack/nack surface becomes an entrypoint; the supervisor pumps the queue. Each message is one bounded step; the unboundedness is the pump.
  3. Timers / cron — sleepUntil generalized to declared schedules in the contract.
  4. main() / full FSM app — the supervised transition-table model of §3.1, potentially co-resident with HTTP handlers and WebSocket lifecycles in one binary.

Pipeline changes: handler_contract.zig's single-entry discovery grows into a multi-entry contract (per-entrypoint routes/capabilities/properties plus an app-level section for the transition table and state schema); precompile.zig embeds multiple bytecode units with an entry table; runtime_cli.zig dispatches on entry kind. The output remains one zero-dependency attested binary. Ship the supervisor behind a -Dapps flag initially (mirroring -Dstudio/-Dedge) so the FaaS binary's threat surface is unchanged while the model matures.

3.3 Capabilities for apps

ModuleCapability already names filesystem and network — but only as vocabulary: today those members are module-internal governance metadata that do not affect handler-level effects or RuntimePolicy derivation. App capabilities are therefore a new authority system, not a mostly-built one, and the enum is the least of it. The design owes: scope normalization, path and port authority, symlink/TOCTOU handling, propagation through helper calls, runtime enforcement, and receipt binding. On that foundation the grants become real, Effects-typed: scoped filesystem path-sets (flow analysis forbids a secret label reaching a filesystem sink outside scope), declared listen ports (the runtime binds; user code never touches raw sockets), a typed persistent_state schema with a size envelope persisted via the durable journal, and onShutdown as a proven step with a bounded finite-step obligation — graceful shutdown as a theorem, not a convention.

3.4 What a proven app gets that hand-written Go/Rust/Node never will

Capability containment machine-checked at compile time rather than approximated by seccomp/containers at runtime. Exhaustive event coverage — every declared (state, event) pair provably handled, every failure fork enumerated. Per-event cost envelopes (Cost Contracts). Behavioral-equivalence across edits including FSM diffs (an added state reads additive; a changed transition reads breaking) before deploy. An attestable binary whose receipt carries the provenance DAG down to the pattern capsules that authored it. Deterministic replay of any journaled incident. No mainstream toolchain offers even two of these.

4. The roadmap

Sequencing logic: NOW makes the reward sound and starts data collection — the flywheel must begin turning before the model exists. NEXT builds the model on data NOW banked. LATER builds standalone foundations on proof machinery (finite-data interlocking, multi-entry contracts) that NOW and NEXT already hardened. HORIZON composes all three. Timeframes are ambition, not commitment — the NOW phase in particular is deliberately over-scoped below its realistic capacity; see the sequencing risk.

6. Risks & open questions

The honest hard parts, after two independent adversarial pressure-tests against the load-bearing claims (a Fable pass and a Codex gpt-5.6-sol cross-check; both confirmed the taint CRIT is already fixed, and their findings are folded in throughout). Where they converged — the reward is gameable by vacuous satisfaction and under-specification, the durable analogy is a narrow precedent, finite tables are not finite-graph model-checking once guards read unbounded state — treat the point as load-bearing.

Reward soundness is a process, not a one-time fix The specific taint fail-open that motivated this concern is closed on main (2026-07-10). The durable point stands and is stronger as a process: a fail-open verifier labels leaky programs green, poisoning both SFT labels and RLVR reward, so the pipeline must version the verifier against a reward-environment manifest (analyzer artifact SHA + soundness epoch — policyHash was recorded unchanged across the July fix, so it cannot gate this) and relabel banked data whenever a soundness fix lands. This is why N1 is data-integrity, not bug-closing. It also means the reward channel is "deterministic with a versioned soundness history," never "perfect."

Does a small local model actually beat a frontier model here?

On the narrow classes — hole fill, repair, form parameterization, canonical transforms — the structural argument is strong and cheaply testable via X2's A/B harness before committing to RLVR. It will lose on novel design, ambiguous requirements, and cross-domain reasoning; the system is architected so losing there does not matter (classify() routes those to the frontier teacher). The real unknown is the middle band — tasks too novel for the small model but too frequent to remote-route economically. Measure the band before concluding.

Does RLVR-on-proofs converge, or reward-hack?

The compiler is unusually hard to game: the proof is total, deterministic, and comptime-anchored, so there is no learned-reward-model exploit surface. But the shaping can be gamed — a policy can learn to emit trivially provable programs that ignore the task. The defense is structural: reward = proof (safety, contract) AND functional tests AND intent assertions; canary tasks whose only correct solutions are non-trivial; periodic human audit of high-reward rollouts. Proof certifies safety; it does not certify intent — conflating them is the single most likely way to waste a training run (the intent-oracle-vs-Spec line already documented internally is exactly this). And the reward is sparse: binary green over multi-line generation is a known-hard RL setting, which is why the per-property discharge shaping and functional partial credit matter, and why convergence is a real risk the A/B gate must clear first.

The decidability tax on standalone apps

The FSM model is more honest than callbacks but less familiar. If developers must hand-author transition tables, adoption stalls. The bet: certified app forms plus agent-authored tables means most users experience "describe the app, review the state diagram." If that bet fails, the fallback is to constrain scope to the entrypoints that need no visible FSM (cli, worker, cron) — which already cover a large share of real solo-dev applications. Separately, acyclic control flow does not bound memory or effects; the persistent_state envelope and journaled-effect seams (§3.1 caveat) are obligations to build, not guarantees to inherit.

Data cold-start

Mitigated for the syntax and repair curricula by a property no other project has: the verifier generates and labels synthetic data with no model in the loop (enumerate canonical forms, mutate, verify; reverse-diagnostics for repair). But near-zero applies only there — synthetic data teaches form, not intent, and a thousand mutations of a few templates are not a thousand independent intents (the committed codegen corpus is ~11 task cases today). Hole-fill and whole-handler training need a minimum count of unique, held-out, acceptance-tested intent families from the teacher or real users before they start — so for intent data the frontier teacher and real usage remain load-bearing, and the cold-start is real.

Where the human stays in the loop

Permanently at three points: intent (task specs and assertions remain human-authored, outside the proof boundary by design); judgment-class decisions the repair-totality classification explicitly reserves; and the approval diff, with the existing veto-exhaustion off-ramp. The trajectory is not "remove the human" but "make every human touch targeted" — provenance DAGs and equivalence verdicts tell the developer precisely which ~5% of an edit deserves eyes.

Underweighted questions to decide early

  • Evaluation against the field. The subset is proprietary, so SWE-bench-style comparisons do not apply — simultaneously the moat and an evaluation vacuum. Define the public metric as cost-per-verified-deployed-handler and first-draft-correct rate versus Claude Code / Cursor driving the same zttp tasks through the plain CLI, and commit to publishing it.
  • Model/spec versioning. The language and proof spec move (policy-hash-relevant surface changes roughly weekly); the local model is trained against a snapshot. Pin the model to a spec-hash and treat a spec bump as a retrain-or-fallback trigger, or the model silently drifts out of distribution.
  • Fork-maintenance cost. A fine-tuned open base model is a dependency to track, patch, and re-license; budget it as an ongoing cost, not a one-time train.
  • The local-model business tension. Local inference removes any per-call model revenue while adding distribution, update, hardware-support, and fine-tune-maintenance cost. Resolve it in one line: if the moat is verifier + data flywheel, shipping weights is fine and monetization sits in deployment, enterprise proof policy, hosted attestation, or support; if the moat is the model itself, "local" contradicts it. Pick one before N5.
  • Addressable-market cap. Restricting to a subset caps what can be built; the standalone-app expansion is partly a market-expansion move, and its scope (which app shapes are in vs out) is a product decision, not only a technical one.
  • Community corpus. Whether the local-first privacy stance permits an opt-in, receipt-anonymized shared corpus — which would supercharge the flywheel — is worth deciding before N5 sets the default.
The thesis in one line zttp expert becomes first-in-class not by having the best model or the best prompts, but by being the only system where the language was shaped for the proof, the proof was shaped to be a reward, and the agent was shaped to author inside the fence — so every hour of use makes the model smarter, every model gain makes proofs cheaper to reach, and every proof deepening gives the agent new vocabulary. Handlers are the beachhead; "the loop lives in the runtime, the logic stays cost-bounded" is the bridge; standalone applications — proven, attested, authored by a local model against a compiler whose reward is deterministic and whose soundness is versioned — are what "agent-compiler" will come to mean.
