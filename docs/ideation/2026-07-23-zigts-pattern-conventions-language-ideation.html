Grounding Context

The thesis this must serve. STRATEGY.md: developers let AI write serverless handlers but cannot know the generated code is correct before shipping. zigttp restricts the JS surface on purpose so correctness becomes decidable, puts the AI agent inside the proof boundary, and the compiler vouches for what the agent writes. Persona: a solo developer with no PM, QA, or SRE. North-star metrics: expert success rate, round-trips to first green proof, proven-path ratio, cold-start p50. One-liner: "The AI writes your handler. The compiler proves it's safe." Every idea below is judged against that loop.

The language today. zigts is ES5 plus arrow functions, template literals, destructuring, spread, finite for-of, optional chaining, nullish coalescing, a match expression, an assert statement with x is T guards, and a pipe operator. It bans classes, this, new, async/await, Promises, var, while, switch, try/catch, null, regex, ==, and ++ — each ban rejected at parse time with a suggestion, and each traded for a proof it unlocks.

The composition base to build on. zigttp:compose ships guard() and pipe() as comptime-only constructs: the parser desugars a |> b and guard chains into a flat arrow function the verifier fully understands (parser/parse.zig ~3320-3355). zigttp:scope handles lifecycle; zigttp:io gives bounded parallel/race; Result<T> is enforced (.ok before .value); the router matches flat "METHOD /path" keys with no groups, coercion, or nesting. Twenty-three virtual modules register at comptime (a Zig file plus a JSON spec each); third-party extensions ride the zigttp-sdk C-ABI into a zigttp-ext: namespace whose distribution and versioning story is unspecified.

The proof machinery already in place. Type capsules Spec<S>, Proof<T,S>, Effects<T,S> are phantom intersection markers stripped before value type-checking — the house rule is to encode every compiler primitive as a TS type, never a new keyword. Canonical normal form (normalize auto-fixer, ZTS612-620) yields an attestable canonical property. Every compile/build/deploy signs contract, bytecode, and rule hashes with a persistent Ed25519 identity and embeds the receipt. The rule registry is comptime-derived so it cannot drift from the checkers; typed RepairIntents already cover ~44 of 71 registry entries. A Law union (pure/idempotent_call/inverse_of/absorbing) exists on Zig module bindings and is discharged by a z3 SMT layer over value-level IR nodes. fault_coverage.zig already enumerates every success/failure fork. Relative TS imports already resolve (file_resolver.zig).

External prior art that shaped the survivors. Roc platforms, Go stdlib minimalism, Deno std, Elm's cohesion-by-refusal, and Unison's content-addressed code all converge on "small core plus capability-scoped modules beats a big stdlib." Composability without classes is a solved problem via higher-order combinators and explicit effects (parser-combinator taxonomies, the Elm Architecture, algebraic effects, Railway-Oriented Programming). Ur/Web is the deepest match for the whole thesis: a type system that proves the absence of whole bug classes, with metaprogramming that generates typed sub-apps inheriting those proofs. Hazel — and the 2024 Blinn et al. work on statically contextualizing LLM completion with typed holes — is the direct precedent for the agent-inside-the-boundary loop.

1

Typed holes — the compiler writes the agent's prompt

language-dxConfidence High · 80%Complexity Medium
Description

Add a hole() (or todo("desc")) builtin — a function call, not a keyword, in the same family as range() and comptime(). A hole parses, type-checks against its expected type, and lets the rest of the program verify normally; at runtime it compiles to a 501 so partial handlers still run under dev. zigttp check --holes --json emits, per hole: the expected type, the in-scope bindings with their types, the remaining capability budget from the Effects ceiling, the undischarged Spec obligations flowing through it, and the module exports whose signatures fit. That JSON block is a statically contextualized prompt for the expert agent — it fills one hole at a time instead of regenerating a whole file.

Basis
external: Hazel typed holes and Blinn et al., "Statically Contextualizing Large Language Models with Typed Holes" (OOPSLA 2024) — the exact architecture of feeding a hole's type and context to an LLM. direct: verification is a tree walk, not a fixpoint (docs/verification.md:26-27), so a typed hole is a leaf that adds no back-edge and stays sound to skip-and-oblige; range() in builtins/number.zig shows the exact builtin-add path; a hole is a call, so the "encode as types/builtins, never keywords" rule holds.
Rationale

Every north-star metric moves at once. Round-trips to first green proof drop because the agent's prompt is exact instead of inferred (project memory records that the real cause of one expert budget-exhaustion was missing module signatures — precisely the context a hole report carries). Expert success rate rises because the expected type constrains generation. And no competing AI-codegen product can hand its model a machine-checked completion contract, because none owns the whole analyzer.

Downsides

The hole report is only as good as the type checker's contextual precision; a hole in a deeply generic position may under-specify. Runtime 501 semantics need a clear "holes present" signal so a holey handler never silently deploys. The IDE/agent JSON schema becomes a stability surface to maintain.

2

Unforgeable Validated<T> brands — parse, don't validate, enforced by type

type-proof-encodingConfidence High · 80%Complexity Medium
Description

Connect three systems that already exist but never talk: nominal branding, validateJson schemas, and flow-analysis labels. Define Validated<T, "schema"> as a nominal phantom capsule (same encoding as Spec/Effects) that only the zigttp:validate return path can construct — unforgeable because nominal types reject structural forgery. Egress sinks (sqlExec, fetch, Response.json on outbound paths) then require the brand in their signatures. Optionally, derive the record type itself from the schema literal the way Pick/Omit already compute types from literals, so one schema definition yields both the shape and the brand. The taint rule ZTS407 stops being a post-hoc flow diagnostic and becomes a local type obligation the agent satisfies incrementally.

Basis
direct: addNominalAlias at type_pool.zig:294 with the comment "prevents structural forgery" (:119); .return_labels = .{ .validated = true } on the validate binding (security/validate.zig:25/38/51); the unvalidated_input_in_egress → ZTS407 rule (flow_checker.zig:64/1396). Project memory records the taint-laundering fail-open CRIT is still open after many review passes. external: Refinement Types for TypeScript (Vekris/Cosman/Jhala, PLDI 2016) layer refinements via intersections with no new syntax; the object-capability model treats capabilities as unforgeable values.
Rationale

"Parse, don't validate" is the strongest known convention for correct handler code; this makes it enforced, not aspirational. It gives the pattern layer (idea 5) a way to demand safe inputs by signature, and it is a second, independent line of defense against exactly the taint-laundering class that a single flow pass has repeatedly failed to close. The agent gets an unambiguous target: the type error names the validator to insert.

Downsides

Retrofitting sink signatures to demand the brand is a breaking change to module surfaces that needs a migration. Schema-to-record-type derivation is a real type-checker feature, not free. A "phase/typestate" generalization (branding a Request as it advances through guards) is tempting but the checker has no flow-sensitive rebinding today — treat that as a later, harder extension, not part of the first cut.

3

Combinator algebra — named guard stacks, route groups, provable coverage

composition-ergonomicsConfidence High · 82%Complexity Medium
Description

Generalize the one trick the repo already proves — that guard()/pipe() are parser-desugared, not runtime values — into a deliberate combinator taxonomy. Add stack(name, ...guards) for named, reusable chains that inline at use sites; support route groups ({ "/api": { use: [guard(auth)], "GET /todos": list } }) that flatten at parse time into the existing "METHOD /path" table with the guard chain woven into every handler in the group; and fold in a rail(f1, f2, f3) combinator that desugars a Result-returning sequence into the .ok-check early-return chain the verifier already demands. Because the desugar sees the whole group, the verifier can discharge a new Spec<"guarded">: "every route under /admin passes requireAuth before its body." Runtime shape is byte-identical to hand-written code.

Basis
direct: the desugar site already synthesizes the flat arrow and enforces "guard composition allows exactly one handler" (parser/parse.zig ~3320-3355); the router is confirmed flat "METHOD /path" with up to 8 params and no groups (modules/src/http/router.zig:73-105); the composition dossier's friction list names "no named middleware stacks, no route groups, no parameterized guards." external: parser-combinator design-pattern taxonomies and Railway-Oriented Programming as the established vocabulary for the rail half.
Rationale

The most-repeated boilerplate in a multi-route handler is re-declaring the same guard chain per route — and the highest-severity bug an AI can generate is the one route that forgot the auth guard. No mainstream framework can prove middleware coverage; zigttp's comptime desugar plus verifier can, with zero new keywords and zero runtime cost. It also gives the agent one canonical spelling to learn, which is where convention actually forms.

Downsides

Desugaring nested groups into the flat table must keep the contract extractor's route list exact, or attestation drifts. Parameterized guards like guard(requireRole("admin")) desugar to a per-request call, not true partial application — fine, but do not oversell it. Group semantics (ordering, override, nested groups) need a spec before they sprawl.

4

Route-table determinism proof — interlocking for routes

type-proof-encodingConfidence High · 85%Complexity Low
Description

Railway interlocking proves, at design time over a finite table, that no two conflicting routes can be set at once. The zigts route grammar is finite and regex-free — literal segments, :param, a terminal *, no nesting — so pairwise overlap, wildcard shadowing, and method-pin completeness are all decidable by construction. Add a compiler pass that proves the route table deterministic (no request matches two routes, no route is dead behind a wildcard, every declared method is reachable) and emit a routes_deterministic contract property, dischargeable as Spec<"routes_total"> and visible in the contract diff.

Basis
direct: matchRoute (modules/src/http/router.zig:73-105) splits on /, matches : params and a terminal *, no regex — a grammar small enough for exact pairwise intersection. Project memory records a shipped AOT method-dispatch bug (2026-07-03) where a match arm dropped the method pin — exactly the conflict class this pass catches. external: railway interlocking / control tables: finite enumerable routes, conflicts fatal, runtime discipline insufficient, so safety moves to a design-time proven table.
Rationale

The proven-route-table pre-filter already exists at runtime; this makes its correctness a proof rather than an assumption, and it catches a defect class the project has already shipped once. It is the smallest sound idea here — a bounded pass over a finite table — and it produces a clean, demo-able attestable property.

Downsides

Low ceiling: it proves one property and does not compound much on its own. Overlap semantics need a decision on whether some overlaps are intentional (a specific route plus a catch-all) and should be allowed with an explicit precedence declaration rather than rejected.

5

Proof-carrying pattern capsules — content-addressed, re-verified on import

pattern-stdlib-layerConfidence Medium · 65%Complexity High
Description

Create a distribution format for higher-layer patterns written in the subset itself: a pattern file exports functions plus their declared Spec/Effects capsules. Its identity is the SHA-256 of its canonical normal form — because normalize produces one stable spelling per operation, two authors who write the same pattern get the same hash, Unison-style, with no registry server. A handler imports it (a zigttp-pat: specifier tier or a local path); at compile time the pattern source is inlined pre-parse and re-verified, so proofs are inherited by re-checking, not by trust — the Ed25519 receipt exists to make the pattern auditable and shareable, not to skip verification. Start with the cheap, shippable-in-days first step: a local zigttp patterns [--json] catalog (the seven examples/patterns/ files promoted to versioned entries, folded into the policy hash) that the expert agent can cite in veto messages. The full self-mining "distill patterns from a corpus of green handlers" idea is a natural sequel once distribution exists.

Basis
direct: extension_bindings.zig is literally an empty binding list (no third-party distribution story); the canonical proof property exists (docs/canonical-profile.md:44-46); Ed25519 receipts ship default-on; relative TS imports already resolve (modules/internal/file_resolver.zig probes .ts/.tsx/.js/.jsx), so "import re-verifies locally" has a real substrate. external: Unison's content-addressed code (identity over a canonicalized form) and Deno std's per-module audit + uniform namespace.
Rationale

This is the strategic bet: it extends the whole thesis from "the compiler proves your handler" to "the compiler proves the shared code your handler stands on," and it does so without a registry, a package manager, or any trust infrastructure — which the solo-dev persona and the zero-dependency constraint both demand. It also turns the expert agent into a pattern author: successful handlers become minted, verified, shareable capsules, which is the compounding flywheel behind the entire ask.

Downsides

The largest surface here by far — specifier tier, hash resolution, receipt verification, catalog tooling. Hash-to-source resolution must stay offline-first (vendored source verified against the pinned hash at compile time); if it ever needs a network fetch it violates the constraints. The flat global module namespace is not uniqueness-enforced, so capsule naming needs a collision story. Sequence it: local catalog first, full content-addressed distribution later.

6

Certified handler forms — comptime generators that inherit proofs

novel-capabilitiesConfidence Medium · 60%Complexity High
Description

Form-based zoning permits any building conforming to a certified form "by right," with no case-by-case review. Ship a small set of certified handler forms — resource({...}), webhook({...}), proxy({...}) — as comptime-only desugars (the exact mechanism guard/pipe use today). The parser expands a declarative literal into canonical handler source — routes, validation calls that mint Validated brands (idea 2), Result rails, and for webhook an unforgettable signature-verify step — and that generated source flows through the ordinary stripper → parser → type-checker → verifier → flow-checker → normalize pipeline. Every generated line is proven exactly like hand-written code because it never bypasses the pipeline; the expansion is committed to a visible file by normalize --write so there is no hidden code.

Basis
direct: compose.zig is .comptime_only = true with stub implementations the parser replaces — the desugar-then-re-prove mechanism is proven in-tree; prior resource() HAL/HTMX work (project memory) shows demand for the declarative shape. external: Ur/Web — metaprogramming that generates whole typed sub-apps inheriting the framework's proofs, named as the deepest match for zigttp's thesis; form-based zoning codes as the "certified form permitted by right" structural analogy.
Rationale

This answers the "reusable components" half of the ask at full strength: a component is not a snippet, it is a generator whose every instantiation is re-proven. For the ~80% of handlers that are CRUD, webhook, or proxy, the AI fills a form instead of free-writing, and green proof becomes near-instant because the envelope was proven once. Because the output is canonical by construction, its hash lands in the receipt — attestable codegen, which no runtime offers.

Downsides

Highest cost and highest ceiling. The form catalog is a design surface that can sprawl; each form's proof envelope must be specified and defended. It naturally sequences after idea 3 (route groups/stacks are its substrate) and idea 2 (brands are what its validation steps mint). Note: this is the buildable, pre-parse form of generation — the tempting "let comptime call user functions gated on a totality proof" version is infeasible because comptime runs before parse/verification exist.

7

Compositional proof provenance — receipts that name the parts

type-proof-encodingConfidence Medium · 65%Complexity Medium
Description

Function-level capsules already compose — a helper typed digest(s): Effects<string, "crypto"> lifts into handler(req): Effects<Response, "crypto">, and the type env keeps spec and effect extractions separate so both stack. What is missing is surfacing the composition: today the receipt attests the handler's final properties as a flat list, discarding the fact that they were discharged by named, reusable functions. Record the discharge tree — "handler is deterministic because paginate is Proof<_, "total"> and formatIso is pure" — into the contract and the JWS receipt, and render it in proof traces. On a failure, the tree localizes which callee broke the chain instead of pointing at the whole handler.

Basis
direct: the two-capsule composition is real — type_env.zig:44-47: "Distinct marker fields let a return type carry both capsules and have each extraction recover only its own," and the effects-capsule.ts example shows two-level composition working today. reasoned: the discharge tree is information the type checker already computes while resolving capsules; persisting it into the existing contract + receipt is a serialization extension, not a new analysis, and it compounds the shipped "proof-explained failures" investment.
Rationale

This turns the pattern layer into visible proof leverage: a user who adopts a proven pattern sees their receipt strengthened by name — "the compiler proves it's safe, and here are the proven parts it stands on," which is the marketing loop for the whole thesis. It also gives the agent a repair heuristic: when a spec fails to discharge, the provenance tree names the callee that broke it rather than emitting a whole-handler error.

Downsides

Modest standalone value — it is most compelling once ideas 5 and 6 exist to produce named, reusable parts worth citing. The receipt schema grows and must stay stable. Provenance depth needs a cap so a deeply nested handler does not produce an unreadable tree.

Cross-cutting combinations

The AI authoring loop, rebuilt — idea 6 gives skeletons (certified forms), idea 1 gives a per-gap completion contract (typed holes), and route/guard coverage from idea 3 is proven as the form expands. Together they attack "round-trips to first green proof" from three sides at once: fewer things to write, exact context for each gap, and coverage guaranteed by construction.
Type capsules as the universal proof carrier — ideas 2 and 7 make the phantom-intersection encoding do double duty: Validated<T> brands carry taint proofs into signatures, and provenance records which branded/proven parts discharged the handler claim. Patterns become safe-by-signature, and the receipt shows why.
A self-strengthening pattern layer — idea 5's content-addressed capsules, once they exist, make idea 7's provenance citations portable across projects (a pattern proven once is cached by content hash), and the "distill patterns from proven handlers" sequel turns every shipped handler into catalog input. Each pattern shipped becomes a reusable proof.
