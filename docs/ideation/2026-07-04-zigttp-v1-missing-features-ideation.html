What zigttp is missing to ship a real app

Seven features scoped to v1.0.0, the rest deferred to v1.1.0. Framed by STRATEGY.md: expert-first, solo-dev persona, hosted cloud deploy is an explicit non-goal.

2026-07-04 · grounded in the v0.1.1-beta tree · lens chosen: ship-a-real-app

The fork this resolves. The marketing names four deploy targets (AWS / Azure / Cloudflare / edge) and none have a working end-to-end path, while strategy says hosted deploy is a non-goal. So "what does 1.0 mean?" was genuinely open. The answer taken here: 1.0 is the point where a solo dev can build → run → deploy something real on their own infrastructure — not a hosted platform.

Design principle. Every v1.0.0 feature keeps a proof-native twist (quota-proven blobs, capability-verified cron, upgrade-proof deploys) so closing the runtime gaps still deepens the moat instead of turning zigttp into a slower Node.

v1.0.0 — the seven

1

A real place to run it

One flag turns the existing static binary into a scratch OCI image; a bundle covers the $5 VPS.

Add deploy --target container (a FROM-scratch OCI image, single-digit MB, ms cold start) and deploy --bundle (systemd unit + Caddyfile for auto-HTTPS + .env template + one-page RUN.md). Retire the dead SAM/wrangler renderers. Runs unmodified on Fly / Railway / Coolify / k8s / docker run — one universal target instead of four broken ones, with zero cloud integration and no control plane (stays inside the non-goal).

Effort: S–M Moves: expert success · retention Proof-native: artifact carries its embedded receipt

build_command.zig · runtime_cli.zig · deploy_manifest.zig

2

Durable single-node state — kv (+ blob)

State that survives a restart, and a place for uploads to land — without leaving the single binary.

A zigttp:kv module (cache API shape, SQLite-backed: get/set/ttl/atomic-incr) plus a zigttp:blob content-addressed local store with streaming writes and range serving. Single-node by design — exactly what the self-host deploy story promises. Cache-lost-on-restart evaporates sessions/counters/rate-limits on every deploy today; uploads have literally nowhere to go (SQL BLOB unsupported).

Effort: kv S · blob M Moves: real-app viability Proof-native: manifest-declared, compiler-enforced blob quotas

modules/src/data/sql.zig · builtin_modules.zig · module_binding.zig

3

cron — capability-verified scheduled triggers

Periodic jobs as a first-class, proof-checked entry point.

cron.every("15m", handler) / cron-expression schedules declared statically; the scheduler lives inside the deployed single binary. This is the only missing capability with no in-runtime workaround — with HTTP + WebSocket as the sole entry points, cleanup jobs, digests, retries, and expiries are impossible. Extends the existing durable_scheduler.zig rather than inventing one.

Effort: M Moves: real-app viability Proof-native: schedules + capabilities verified at compile time

durable_scheduler.zig · runtime_workflow.zig · builtin_modules.zig

4

Ops surface — /metrics, /healthz, request-ids, error webhook

A deployed handler stops being a black box.

Promote metrics out of Debug-only into release builds: a Prometheus-text /metrics (per-route latency histogram + status counters), /healthz, a runtime-generated request-id echoed in a response header and every log line, and a 5xx onError webhook as the poor-man's error reporter. The persona has no SRE — the runtime must be its own. Also makes the cold-start p50 guardrail observable outside the bench harness.

Effort: M Moves: operability · cold-start guardrail

server.zig · runtime_pool.zig · contract_runtime.zig

5

Proof-explained failures

Every runtime error names its .ts line and its relationship to the proof.

Populate the existing (currently-null) source-map emission so faults report file/line + a stack. Classify each fault against the contract ("occurred on a path marked unproven for reason X"). The sharp edge: a fault on a proven path is auto-captured as a soundness incident with a minimized repro into the witness corpus — by definition a compiler bug. Turns the missing-debugger weakness into the strongest claim: proven means it cannot happen, and we put that in the error handler.

Effort: M Moves: debuggability · trust Proof-native: fault → proof-chip mapping

interpreter.zig · compiler.zig · zruntime.zig · fault_coverage.zig

6

Upgrade-proof on deploy

deploy tells you whether the new binary behaves differently — before you swap it.

Wire the already-built prove_upgrade.zig / upgrade_verifier.zig + capsule replay (currently used by prove/rollout, not deploy) into the deploy flow: diff the new artifact's proven route behavior against the previous deploy's capsule and print "compatible / N routes changed behavior — here's the counterexample." The persona has no staging and no QA; the scariest moment is redeploying over a working app. The most differentiated idea here — nothing generic can copy it.

Effort: M Moves: deploy safety · retention Proof-native: private deploy-time behavioral diff

build_command.zig · prove_upgrade.zig · upgrade_verifier.zig · capsule.zig · proof-review/review.zig

7

Real-app onboarding pack

The paved road from init to a deployed, proven CRUD-with-auth app.

Two init templates (crud = REST + sql + validate; auth = sessions + signup/reset), a "your first REST API, proven and deployed" tutorial ending at deploy --target container, and an Express/Hono/Workers migration guide whose core is a restriction-mapping table ("you reached for try/catch — here's the proven equivalent, and here's what the compiler now guarantees"). Today the path is one durable-workflow tutorial plus unguided examples.

Effort: M (docs + templates) Moves: round-trips-to-green · 2nd-handler retention

init_command.zig · cli_templates.zig · docs/tutorials/ · examples/

v1.1.0 — deferred (and why)

These survived the critique but belong past 1.0. The starred item is the strongest 1.1 candidate.

  • 1.1 leadMCP proof oracle + per-tool rules-packs (init --agent). The highest-leverage idea overall: puts zigttp's proofs inside Cursor / Claude Code / Codex — a distribution channel no generic runtime can copy — and un-defers a bet you already won. Low effort (the pi-free analyzer, JSON diagnostics, and WASM surface all exist to wrap; only one incidental mcp mention in the tree today, so it is build-not-flip). Deferred only because it is the proof-system play, not ship-real-app.
  • 1.1Shareable signed proof card + consent-native retention metric. Un-defers the trust artifact through the right persona (a solo dev bragging) and gives the aspirational second-handler-retention metric its first measurement home. A growth play, not a shipping prerequisite.
  • discipline"1.0 = frozen proof surface." Adopt the versioning discipline now — name the subset grammar + proof semantics in every receipt — but it is a framing move, not a shippable feature.
  • 1.1Subset LSP. The biggest DX friction for hand-editing, but expert-first means the agent is the main author; L effort; the 1.1 rules-pack covers ~80% of the need for the agent.
  • 1.1Provable bounded-regex subset. The most on-thesis large bet, but the highest engineering cost, and extended validate formats already cover ~70% of real need.
  • cutProof-edge test generation. Shipping a test generator next to "the compiler proves it" invites "so proofs aren't enough?" — strategic tension.
  • cutWitness-memory agent. Speculative gains; the corpus is too thin at launch to retrieve anything useful.
  • cutNative inbound TLS. Security-sensitive surface for marginal gain over the Caddy recipe in feature 1. Ship the recipe; cut native TLS.
  • cutFull proof-sealed-secrets flow analysis ("key flows only to api.stripe.com"). Risks over-claiming — a research project. The shippable subset (name-level capability + deploy injection) folds into features 1 and 4.
  • cutJudge-mode playground / proven-handler corpus. Drag toward parsing arbitrary JS (fidelity non-goal) and platform gravity (team-persona non-goal).
  • fold-inNormalize-complete (canonical auto-fix). A real papercut but narrow hygiene; fold into proof-engine polish rather than a headline feature.

Recommended first move

Take feature 1 (a real place to run it) into ce-brainstorm first — it is the load-bearing "where does it run?" answer, it unblocks the retention metric (a handler with no URL never earns a second), and it retires the incoherent four-targets marketing.

Tighter alternative: brainstorm 1 + 4 + 6 as one "deploy & operate a real handler" slice — they share the deploy surface and together turn "it built locally" into "it runs, it's observable, and it's safe to redeploy."

Coherence fix

Update the "targets AWS/Azure/CF/edge" marketing to "runs anywhere a container or a file runs" — honest, and stronger.

Scope guardrail

No hosted control plane, no cloud credentials, no phone-home. Every v1.0 feature runs on the dev's own infrastructure.

Method: three read-only codebase scouts → surface decomposition → three divergent ideation lenses (~30 candidates) → critique to survivors. Companion detail in the ideation working notes. Next: ce-brainstorm on the chosen feature to turn it into requirements.
