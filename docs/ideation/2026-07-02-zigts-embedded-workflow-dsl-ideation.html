Ideation: ZigTS Embedded Workflow DSL

  • Date: 2026-07-02
  • Topic: zigts-embedded-workflow-dsl
  • Focus: minimal durable, fault-tolerant workflow example and DX prose
  • Mode: repo-grounded

The strongest direction is not a new API. It is a tiny embedded workflow grammar built from names, durable boundaries, proof receipts, and operator-visible artifacts. The example should let a user see one durable run key start, dispatch one queued child, survive one fault, and produce one proof receipt.

Grounding Context

Codebase Context

Zigttp already has the durable workflow runtime pieces: durable runs and steps, timeout and signal primitives, top-level workflow dispatch, persisted queued child dispatch, dead letters, parent suspension and replay, and compile-time diagnostics for invalid durable shapes. Existing docs and examples cover the primitives, but the first-path mental model is scattered across tutorials, proof docs, queue docs, and separate workflow fixtures.

The example should stay inside today's ZigTS surface. That means no new workflow API, no implied async promises, no hidden actor state, and no conflation between the persisted workflow queue and the separate actor mailbox queue.

1. One Run Key, One Child, One Recovery

Confidence 90% Complexity Medium
Description
Make the canonical example a small workflow: derive a run key from Idempotency-Key, enter durable.run(key), dispatch exactly one top-level workflow.call through --workflow-queue, inspect one recovery artifact, and end with one proof receipt.
Axis
Runnable workflow story.
Basis
direct: examples/workflow/queued-orchestrator.ts already shows run(key) and top-level call; docs/tutorials/first-durable-workflow.md already teaches durable start, queued child dispatch, signal/crash drills, and proof checks.
Rationale
Users need one copyable shape before they need a catalog of primitives. This makes the embedded DSL feel real without adding syntax.
Downsides
It under-shows wait signals, saga, and fanout on purpose. Those should become second-step examples, not the first mental model.

2. Proof Receipt First

Confidence 88% Complexity Low
Description
Put zigttp check --json plus zigttp check --contract, or the verification receipt, before the live server drill. Annotate which line creates the durable run, which line creates queued child dispatch, and which proof field confirms or withholds the behavior.
Axis
Proof and DX feedback.
Basis
direct: docs/proof-card.md, docs/verification.md, and the contract builder already expose durable workflow proof fields and diagnostics.
Rationale
This differentiates ZigTS from generic durable workflow demos. The user sees the compiler's workflow verdict before trusting the runtime story.
Downsides
The receipt needs careful copy so it reads as actionable authoring feedback, not as a wall of JSON.

3. Deterministic Failure Drill

Confidence 85% Complexity Medium
Description
Make one repeatable fault path part of the example: force a queued child into dead letter or parent suspension, inspect it, replay it, and show the parent finishing without duplicating already-recorded work.
Axis
Durability and fault semantics.
Basis
direct: packages/runtime/src/workflow_queue.zig and packages/runtime/src/zruntime.zig implement persisted queue states, dead letters, replay/discard, and parent resolution.
Rationale
Durable behavior is most credible when failure is not abstract. The user should see the durable artifact and the recovery command in the same workflow story.
Downsides
A failure-first path can feel heavier than a hello-world flow. Keep the drill deterministic and one-screen small.

4. ZTS509 Negative Twin

Confidence 86% Complexity Low
Description
Add a tiny adjacent invalid snippet: durable.step("charge", () => workflow.call("payment", ...)). Show ZTS509, then the corrected version with workflow.call at durable depth 0 inside run.
Axis
Proof and DX feedback.
Basis
direct: packages/zigts/src/contract_builder.zig and packages/zigts/src/rule_registry.zig define and explain the durable-depth diagnostic.
Rationale
The most useful DX prose often names the forbidden shape before users discover it. The corrected twin teaches the workflow DSL boundary rule faster than prose warnings.
Downsides
Negative examples must stay visually subordinate so the docs do not lead users into copying the wrong file.

5. Artifact Tree + Fault Matrix

Confidence 83% Complexity Low
Description
Put a compact table next to the example: failure trigger, persisted artifact path, caller-facing response, operator command, and proof or idempotency condition. Use it to explain the durable directory as part of the DSL, not as internals trivia.
Axis
Durability and fault semantics.
Basis
direct: docs/durable-workflows.md, docs/tutorials/first-durable-workflow.md, and runtime queue/dead-run code already expose the artifact and operator model.
Rationale
Fault tolerance becomes easier to debug when each visible file or CLI command maps back to one workflow boundary.
Downsides
The table can drift if artifact paths or CLI names move. Keep it backed by examples or doc-link checks.

6. DSL As Naming Convention

Confidence 82% Complexity Low
Description
Define the embedded DSL as disciplined names and boundaries: run key, step names, child handler names, signal names, idempotency key, timeout names, and forbidden boundaries. This becomes a short pattern header above the example.
Axis
Documentation/prose structure.
Basis
reasoned: the current APIs already expose the durable contract through named boundaries, and adding a new DSL layer would increase surface area before the existing one is learned.
Rationale
This gives the feature a DSL feel while preserving ZigTS's explicit, inspectable style. It also gives humans and agents a small authoring checklist.
Downsides
Naming discipline is softer than compiler syntax. The prose must be crisp and backed by proof output to avoid feeling like convention-only advice.

7. Queue Type Disambiguator

Confidence 84% Complexity Low
Description
Add a small "which queue is this?" section. --workflow-queue is persisted child dispatch for durable workflows. --actor-queue and zigttp:queue are process-local actor mailboxes. fanout should be described as ordered durable batch grouping, not concurrency.
Axis
Documentation/prose structure.
Basis
direct: workflow docs and runtime code define the persisted workflow queue; actor queue notes identify zigttp:queue as a separate opt-in mailbox surface; fanout runtime behavior is declaration-ordered.
Rationale
This prevents users from picking the wrong reliability model and gives the prose a clearer embedded-workflow grammar.
Downsides
It is mostly prose leverage. It should be paired with code comments and fixture names so the distinction appears in commands, not only docs.

Top Pick

Start with idea 1, then fold ideas 2, 3, 4, 5, 6, and 7 into the same artifact as supporting rails: proof first, one deterministic failure drill, one negative twin, one artifact matrix, and one queue-disambiguation note.
