What if restrictions were features?

zigttp removes while loops, classes, try/catch, and var from JavaScript. Not because they are bad, but because removing them makes the IR tree identical to the control flow graph.

When every branch is visible at compile time, the compiler can walk every execution path. It proves that every path returns a Response, that Result values are checked before access, and that no state leaks between requests.

The restrictions are not limitations. They are the mechanism that makes static proof possible.

handler.ts 
import { env } from "zigttp:env";
import { jwtVerify } from "zigttp:auth";

function handler(req: Request): Response {
  const token = req.headers["authorization"];
  const secret = env("JWT_SECRET");

  if (!token) {
    return Response.json({ error: "no token" }, { status: 401 });
  }

  const result = jwtVerify(token, secret);
  if (result.ok) {
    return Response.json({ user: result.value });
  }
  // bug: missing return when verification fails
}

Everything proven at build time

A single build command with -Dverify and -Dcontract produces a contract manifest - a machine-readable proof of what your handler does.

The contract lists every environment variable accessed, every outbound host contacted, every cache namespace used. When the compiler can prove the full set, the handler is automatically sandboxed: it can only access what was proven.

No annotations. No configuration files. The proof comes from the code itself.

Native speed, no npm

Virtual modules replace npm packages. JWT verification, JSON Schema validation, cryptographic hashing, request caching - all implemented in native Zig, callable from JavaScript with zero interpretation overhead.

One binary. Zero supply chain. No node_modules, no lock files, no transitive dependency tree to audit.

The engine itself outperforms QuickJS across every benchmark category, from 63x on string operations to 3.1x on full HTTP handler throughput.

TypeScript and JSX, natively

No tsc. No Babel. No build step. Write .ts and .tsx files directly. The native TypeScript stripper removes type annotations at load time, and the type checker validates them at build time.

Sound mode goes further: it proves that arithmetic operates on numbers, boolean contexts receive values with meaningful falsy states, and discriminated unions are narrowed correctly in match expressions.

JSX compiles to virtual DOM nodes and renders to HTML strings on the server. Full SSR from a single binary.

handler.tsx 
import { routerMatch } from "zigttp:router";

type User = {
  name: string;
  role: "admin" | "member";
};

function UserCard(props: { user: User }): JSX.Element {
  return (
    <article class="card">
      <h2>{props.user.name}</h2>
      <span>{props.user.role}</span>
    </article>
  );
}

function handler(req: Request): Response {
  const route = routerMatch(req.url, ["/user/:id"]);

  if (!route) {
    return Response.text("Not found", { status: 404 });
  }

  const user: User = { name: "Ada", role: "admin" };
  return Response.html(
    renderToString(<UserCard user={user} />)
  );
}

Test by declaring, not mocking

Handlers in zigttp are pure functions of a request and its virtual module responses. There is no hidden state, no database connection, no environment to simulate.

Testing is declaring inputs and expected outputs in JSONL. Each test line specifies a request, the virtual module stubs, and the expected response. No mocking framework. No test infrastructure.

The same system powers deterministic replay: record production traces, replay them against a new handler version, and the build fails if behavior changes.

Get started

Build from source (requires Zig 0.16.0+) 
git clone https://github.com/AurelioSilva/zigttp.git
cd zigttp
zig build -Doptimize=ReleaseFast
Write a handler 
function handler(req: Request): Response {
  return Response.json({ message: "hello", path: req.url });
}
Run it 
zig build run -- handler.ts -p 3000
curl http://localhost:3000/
# {"message":"hello","path":"/"}
